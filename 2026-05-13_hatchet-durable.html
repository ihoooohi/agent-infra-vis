06 · Hatchet Durable Task

为什么 Athena 跑 30 分钟、用户离线 8 小时回来，按钮还能点
Athena 的 7 步 autopilot 经常要跑 30-60 分钟，中间还要等人在 IM platform 上点确认按钮（步骤 3 和 4），人可能 8 小时后才看到。如果是普通 background task：容器一重启 / worker 一挂，整个流程作废。
用 Hatchet durable task 解决：状态持久化、断点续传、event 等待跨重启。代价是要按 Hatchet 的怪规矩写代码 —— 比如不能用动态 event key，得用 固定 type + CEL filter。

1为啥不用普通 background task

对比一下"普通 asyncio task" vs "Hatchet durable_task"在 Athena 场景下的表现：

❌ 普通 asyncio.create_task
最常见的 Python 做法
  • 容器重启 → 任务直接丢
  • worker crash → 同上
  • 等用户点按钮 30 分钟 → 内存里 hold 一个挂起的 coroutine，浪费连接 / FD
  • 没有重试机制
  • 没有跨实例观测
✓ @hatchet.durable_task
Athena 选的方案
  • 状态每个 await 边界都 checkpoint 到 Hatchet 后端
  • 容器重启 → 在原 await 处无缝 resume
  • 等 event 不占进程资源 —— Hatchet 服务端等
  • 失败自动重试（可配 max_retries）
  • 有观测后台 看每个 task 哪步、跑多久

2autopilot 工作流定义

@hatchet.durable_task(
    name="campaign-autopilot",
    input_validator=AutoPilotInput,
    execution_timeout="60m",    # 一次完整运行最长 60 分钟
    schedule_timeout="60m",     # 排队等 worker 最长 60 分钟
)
async def autopilot(input: AutoPilotInput, ctx: DurableContext) -> dict:
    """Main durable task — runs all 7 steps sequentially.
    Uses durable context for human-in-the-loop waits."""
    ...

关键参数：

  • execution_timeout=60m · 整个工作流总执行上限。LLM 慢 + 人工等待都算在里头
  • schedule_timeout=60m · 提交后等 worker 接手的最长时间。worker 全忙时排队
  • input_validator · Pydantic 模型，运行时校验输入
  • DurableContext · 不是普通 Context —— 这是 Hatchet 提供的"可持久化上下文"，每次 await 都做 checkpoint

3核心模式 · 固定 event type + CEL filter

workflow 等用户点确认按钮的代码 —— 这是 Hatchet 最关键的"踩坑笔记"：

❌ BAD — 动态 event key
await ctx.aio_wait_for_event(
    f"human_confirm:{run_id}:step{N}"
)

每个 run / step 一个唯一 key。worker 重启后这个 key 没被重新注册 —— Hatchet 收到 event 找不到接收者，直接丢。用户点了按钮永远没反应。

✓ GOOD — 固定 type + CEL
await ctx.aio_wait_for_event(
    "human_confirm",
    f"input.run_id == '{run_id}' "
    "&& input.step_id == '{N}'"
)

固定 type 在 Hatchet server 端永远是 "known"。CEL filter 在服务端从 event log 里筛出匹配的，重启不影响。

这条规则写在文件顶部注释里： "Use FIXED event type + CEL filter, NOT dynamic event keys." —— 配 ref 链接和踩坑日期。
"Why: Hatchet drops events for unregistered keys. Dynamic keys are unique per run — after worker restart, the key isn't re-registered, and events arrive to nobody."

44 种用户 action

用户在 IM platform 卡片上点按钮，触发 4 种可能的 action。Hatchet event 的 input.action 字段区分：

confirm
"OK，过"。接受当前 step 输出，进入下一 step
reject
"不行，重做"。带 reason 触发 on_step_rejection 生成 reflection，再跑一次本 step
cancel
"算了，停"。剩余 step 全标 error，workflow 直接返回
skip
"这步跳过"。step 标 done，继续下 step（不重跑）

6双源读 · 跨进程的 race 修复

Hatchet 的一个隐藏陷阱 —— 在 async 边界，DurableContext 可能在不同 Python 进程恢复。模块级缓存会消失。

# workflow/hatchet_workflow.py · line 462-488 注释 # Dual-source read (2026-04-15 residual race fix): # Hatchet DurableContext may suspend the worker at async boundaries and # resume in a different Python process. The module-level # _creator_perf_json_cache is process-local and would silently disappear # across such a restart, causing FastAPI to fall through to Nova Micro # extraction → card never renders. # Fast path: same-process module cache (O(1) dict lookup). # Durability path: WorkflowState JSON file (populated at line 826 by # _prefetch_creator_perf, survives worker restarts).

修复方案 = 双源读：

_creator_perf_json_cache 的读取策略
先快后慢 —— 命中快路径就用，没命中再去慢路径
fast path
_creator_perf_json_cache[run_id]
模块级 dict · 同进程 · O(1) 查找
命中 ≈ 99% 的正常情况
→ 没命中 →
durable path

workspace 卷上的 JSON 文件
跨进程 / 跨容器都存在
修复成本： 多写一段 fallback 代码，多一次 IO（仅在 cache miss 时）。问题症状： "卡片永远不渲染"—— 因为 cache 丢了 → FastAPI 拿不到 structured JSON → 走 Nova Micro 抽取 → 失败 → 前端无法识别 schema。

7WorkflowState · workspace 上的状态文件

Hatchet 已经管 task 状态了，为什么还要 WorkflowState？答案在 autopilot() 的 P2 注释里：

# CRASH-SAFE STATE (P2) # WorkflowState persists step results to workspace volume # (survives container restarts). # On Hatchet replay after crash, completed steps are already in the state file. wf_state = WorkflowState() existing_run = wf_state.get_run(run_id) if not existing_run: wf_state.create_run(campaign_id, chat_id, run_id=run_id) wf_state.update_run(run_id, status="running") # Restore step state from file if resuming after crash saved_run = wf_state.get_run(run_id) if saved_run: for saved_step in saved_run.get("steps", []): sid = saved_step["id"] if saved_step.get("status") == "done": steps_state[sid]["status"] = "done" steps_state[sid]["summary"] = saved_step.get("summary")

Hatchet checkpoint 只能粒度到 await 边界。step 中间发生的各种局部状态变化（已发送哪些卡片、已拉哪些数据缓存、已写入 progress.md 哪些段）不会自动持久化。WorkflowState 是这层的补丁：

  • 每个 step 完成时手动写 WorkflowState · 包括 status / summary / step_specific_data
  • Hatchet 重放 task 时（容器重启 → resume）也会从 step 0 重新执行 autopilot()
  • 但 autopilot() 一开始先检查 WorkflowState，已 done 的 step 直接跳过、读取保存的 summary
  • 结果：用户不会看到"step 1 again? again?"的诡异界面
WorkflowState ≠ Hatchet state： 是两层独立的状态。Hatchet 管"task 这一次跑到了哪个 await"；WorkflowState 管"业务上哪些 step 完成了"。两层一起才完整。

9Hatchet worker · 怎么跑起来

worker 在 nanobot_server.py 启动时挂上：

# nanobot_server.py 启动逻辑
hatchet = Hatchet(debug=True)

# Hatchet workflow handler runs on a SEPARATE THREAD
# (asyncio for FastAPI + thread for Hatchet)
_hatchet_worker = ...
_hatchet_worker.start()

关键架构：

  • FastAPI 跑在主 asyncio event loop —— 处理 IM webhook、API 请求
  • Hatchet worker 跑在独立线程 —— 处理 durable_task
  • 两边都改 _active_cards 等共享 dict —— 所以必须 threading.Lock 不是 asyncio.Lock

10关键收获 · Hatchet 不是银弹

Hatchet 给了 durability，但要求开发者：

  1. 所有 event key 必须静态固定 · 用 CEL filter 区分
  2. 不能依赖模块级 / 进程级缓存 · async 边界 = 跨进程边界
  3. 业务层状态自己持久化到 workspace · Hatchet 只管 task 调度状态
  4. terminal state 推送要单独重试逻辑 · 别让 web UI stuck
  5. 与 asyncio 共存时 lock 用 threading.Lock（不是 asyncio.Lock）
