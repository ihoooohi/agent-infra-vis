07 · Auto-Pilot 7 步深扒

每天替团队跑的一整套 TikTok 红人活动管理 —— 每步具体在干什么
Auto-Pilot 是 Athena 的"主业" —— 每天 6:00 / 操作员手动 POST /autopilot 触发，跑完一整套红人活动的诊断 → 决策 → 报告。Step 1-2 / 5-7 是纯自动（约 30-40 分钟），Step 3-4 要人在 IM platform 上点按钮决策（操作员等 1-8 小时）。
设计上有个反复出现的模式："deterministic 算 + LLM 写人话" —— 数字计算交给 Python，措辞 / 解读 / 战略建议交给 Claude。
编号小坑： 代码里 STEPS 数组 id 从 0 开始（0-6），所以 step_id=2 实际是给用户看的 "Step 3/7"。这是 Python list 索引和人类编号的对位差异。本文沿用用户视角编号 1-7。

1Step 1 · Campaign Performance

step_id=0 📊 Campaign Performance AUTO
拉过去 7 天数据，建立 baseline —— 后续步骤都基于它判断"涨还是跌"。
多 dashboard 探测：每个 campaign 可能有 {campaign_id} 主 dashboard + {campaign_id}_auto_agent_april 新创作者群组，必须合并数据再分析。

输出格式（< 300 词执行摘要）：
  1. 总视频数、总播放、总互动
  2. 平台分布（TikTok / IG / YouTube）
  3. 趋势方向（vs 上周）
  4. 多 dashboard 时单独标 breakdown
skills/campaign-performance campaign videos API prev_block (无)

2Step 2 · Top Creators Analysis

step_id=1 🌟 Top Creators Analysis AUTO
挑前 5 名创作者（合并 dashboards 排名），提炼共性，给其他创作者复用。
每个 top creator 输出：
  • handle + 平台
  • 最佳表现的视频（标题、播放、互动率）
  • 什么 hook / format 让它有效
  • 可复制的模式

仅查内部创作者，prompt 明确说 "do NOT search external social media for this step"。
skills/campaign-performance prev_block: Step 1 result

3Step 3 · Underperformer Diagnosis（人工确认）

step_id=2 🔍 Underperformer Diagnosis HUMAN ✋
诊断表现差的创作者，分类 CUT / COACH / ON_TRACK / INSUFFICIENT_DATA，等用户在 IM platform 卡片点 confirm/reject。
这是 Auto-Pilot 最复杂的一步。它有 Mode A / Mode B 双模式：
Mode A · 首选
prompt 里塞了 "=== PRE-COMPUTED CREATOR PERFORMANCE ===" 段。这是后台预算好的结果（_prefetch_creator_perf 在用户等待 Step 1 时悄悄做）。LLM 只需读分类 + 写诊断 / 建议，不用算数。
Mode B · 回退
预算失败（API 超时 / dashboard 不存在）。LLM 自己调 API 拉数据、group by creator、按 policy 阈值分类。严禁瞎编 handle —— 若拿不到数据要明确说 "Data unavailable — strategist review required"。
每个 creator 的分类输出：
CUT
建议放弃。1 行 rationale
COACH
建议辅导。diagnose + 1-2 句具体建议
ON_TRACK
表现符合预期（≥1K avg views），不细写
INSUFFICIENT_DATA
数据不足，待观察
品牌语气约束（写在 prompt 里）： "Be factual and respectful. These are real creators." 不能说 "terrible performance"，应该说 "not meeting the 1K views bar"。这是 Athena 写给真人看的，用词直接代表公司形象。
workflow/creator_performance.py _creator_perf_json_cache creator-performance-policy.md

4Step 4 · Creator Replacement（人工确认）

step_id=3 🔄 Creator Replacement HUMAN ✋
给 Step 3 中被 CUT 的位置找 3 个替补，按严格搜索顺序。
搜索顺序（必须严格遵守 · 读 SKILL 再搜）
1
内部 pool
已签约 / 已合作过的创作者
skills/creator-search
2
44M creator 数据库
未签约但有 profile 的，bio / hashtag 匹配
skills/creator-sourcing
3
跨 campaign 顶级表现者
在其他 campaign 表现好、可能 fit 当前 niche
skills/campaign-performance
4
外部社交媒体爬虫 · 最后兜底
前 3 步还凑不够 3 人才用
social-scraper
筛选标准： 同 niche，10K-500K 粉丝（中尾创作者更好谈），最近活跃。
bonus： 已经在自发提到产品的创作者是首选 —— "organic mention" 比硬广效果好。

每个候选输出：handle、平台、粉丝数、avg views、source（来自哪个 pool）、风格 fit 解释、为什么是好替补。
skills/creator-search skills/creator-sourcing skills/campaign-performance

5Step 5 · Video Review

step_id=4 🎬 Video Review AUTO
审 DRAFT 视频（草稿，未发布），按 video-preferences.md 硬规则 PASS/FAIL 给 3 种 verdict。
这是一个 "二进制判断" 任务 —— 每条规则 PASS 或 FAIL，没有"大概也行"。

数据流：
  1. Gemini vision 后台预分析视频内容（在 Step 4 后台同步跑）
  2. 每个视频对每条硬规则给 PASS / FAIL + evidence
  3. 结果按 schema 注入到 Step 5 prompt
  4. LLM 汇总 + 给最终 verdict
APPROVED
所有硬规则 PASS · 直接发布
REJECTED
任一硬规则 FAIL · 退回创作者
NEEDS_REVIEW
有 SKIPPED 规则（Gemini 没分析出来）· 需人工 review
规则定义在 video_analyzer.py 的 DEFAULT_RULES： rule id 和数量动态从这里读，prompt 里用 {_rule_ids} / {_rule_count} 插入 —— 改规则不用改 prompt，硬编码已经被消除。
workflow/video_analyzer.py _video_analysis_cache _gemini_structured_cache video-preferences.md

6Step 6 · Competitive Intel

step_id=5 📡 Competitive Intel AUTO
扫竞品 + niche 趋势 —— 给 Step 4 找替补和 Step 7 日报当原材料。
完全独立的子系统在 workflow/competitive_intel.py。三种数据源：
A
Exa 搜索 · 竞品市场动作
_search_exa_competitor_marketing(competitor_name) · 找新闻 / 博客 / 社交媒体关于竞品 marketing 的报道
B
Exa 搜索 · niche 战术
_search_exa_niche_tactics(niche) · 该 niche 当前流行的 hook / format / 创作者
C
TikTok API · 视频 + hashtag
_search_videos / _fetch_hashtag_data · 趋势播放量、热门 sound
所有数据汇总成 format_competitive_data() 文本块，注入到 Step 6 prompt + 缓存到 _competitive_intel_cache，Step 7 日报也用得上。
workflow/competitive_intel.py Exa search API TikTok scraper API _competitive_intel_cache

7Step 7 · Daily Report

step_id=6 📝 Daily Report AUTO
把前 6 步的关键信息汇总成日报，发回 IM 群。
prev_block 累积传递 —— Step 7 的 prompt 拼了前 6 步的产出摘要。所以 Step 7 不是重新读数据，而是把已有结论组织成可读日报。

输出内容（典型结构）：
  • 📊 Performance · Step 1 摘要
  • 🌟 Top performers · Step 2 top 5
  • 🔍 Underperformers · Step 3 决策结果（用户 confirm 后的）
  • 🔄 Replacements · Step 4 接受的替补
  • 🎬 Video reviews · Step 5 verdict 统计
  • 📡 Market intel · Step 6 trends
  • 📌 Action items · 明天该做什么
prev_block: Steps 1-6

8Mode A 的预算机制 · 在用户等待时悄悄算

Step 3 / 4 / 5 都用 "预算 + 缓存注入" 模式 —— 重活在用户等待时跑，Step 真正执行时只读结果。这是 Athena 工程上最关键的响应性优化。

1
autopilot() 启动后立即
不等 Step 1，平行启动 _prefetch_creator_perf() · prefetch_competitive_intel() · video_analysis 调用
2
Step 1 / Step 2 跑
用户看到第一步推进，后台预算并行进行
3
预算完成 → 写 3 个 module 级 cache
_creator_perf_json_cache + _creator_perf_text_cache + _video_analysis_cache + _competitive_intel_cache · 同时写 WorkflowState 文件
4
Step 3 prompt 拼 PRE-COMPUTED 段
_creator_perf_text_cache[run_id] 注入；Mode A 生效
双源读保护： 第 6 篇文章讲过 —— Hatchet 在 await 边界可能换进程，模块 cache 丢。所以每个缓存都同时写 WorkflowState 文件，cache miss 时从文件回退。

9用户拒绝时的 reflection 机制

Step 3 / 4 等到 confirm 时还有第 4 个 action：reject + reason。这时不是简单重跑，而是触发 v7 reflection 流程：

1
用户点 Reject + 写原因
"@ 这周才入组，先观察"
2
on_step_rejection() 生成 reflection
LLM 读 step output + 用户 reason，写一段"为什么这次输出不行"的反思
3
reflection 存到 workspace + pgvector
未来同 campaign 跑 Step 3 时这条 reflection 会被 memory.search() 检索到
4
verification_feedback 注入 Step prompt
"OPERATOR REJECTION: ... REFLECTION: ... Please redo this step addressing feedback."
5
step 重新跑
这次 LLM 知道上次为啥被拒，能避开同样错误
这是组织级学习的雏形： 操作员的"拒绝原因"被沉淀进 memory，同一个 campaign 后续运行都会受益。Athena 不是"每天从零开始"，是"每天比昨天聪明一点"。
