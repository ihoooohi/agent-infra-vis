三层调度 · Hatchet / Prefect / cron 撑起"后台 agent"

为什么不是一个调度器搞定所有 · 每层不可替代在哪 · 一个任务该放哪层
一个 agent 平台真正"活"起来，光有对话循环不够 —— 后面还要有主动跑活的肌肉：每天给运营推 24h 花费报告、每周生成竞品分析、Athena 7 步 autopilot 中间挂起等用户点按钮、上百万条达人数据夜里入库……
这些事的性格完全不同：有的需要状态持久化 + 等人 + 跨进程 resume，有的需要批量调度 + worker pool + 失败重试，有的只是"每天 9 点跑一个 Python 脚本"。
用同一个调度器全包，不是技术做不到，是性格不合。所以我们形成了三层：Hatchet 管长事务、Prefect 管离线 pipeline、cron 管宿主小活。

1三层全景 · 各自的位置

先看一眼分工地图。三层不是层级（不是上层调用下层），是按任务性格平行存在的三种"调度肌肉"：

Hatchet长事务 · 等人 · 不丢
agent 业务流的"耐久骨架"。一次 task 可能跑 30-60 分钟、中间挂起等用户点 IM 按钮、容器重启不影响。状态在 Hatchet server 端持久化，每个 await 边界都 checkpoint。
代表场景：Athena 7 步 campaign autopilot · competitive_intel · creator_performance
Prefect批量 pipeline · 数据加工
离线/夜间批处理的"工厂"。flow run 跑爬虫、入库、聚合、生成日报。worker pool 模型 —— 任务被分发到不同 worker 上跑，可以横向扩。失败有重试、有观测后台。
代表场景：达人采集 pipeline · 视频索引 · reflection_agent · IM 群日报聚合
cron宿主小活 · 简单粗暴
EC2 宿主机自己的 crontab + 一个 Python 脚本。没有 worker、没有状态、没有 UI —— 但极其简单可靠，不依赖任何调度服务。Hatchet/Prefect 全挂了它也照跑。
代表场景：amy_cost_daily_report · collect_token_usage · 各种"每天发一张卡片"的脚本
心智模型： Hatchet = "在对话流里挂起，等人，再继续"；Prefect = "大批量数据夜里入库，跑完发报告"；cron = "9 点准时给老板发花费报告"。三件事都叫"调度"，但要的能力完全不同。

2能力矩阵 · 三层都能干什么

把"任务调度"拆成 8 个具体能力，看看每层各自的强弱：

能力维度 Hatchet Prefect cron
状态持久化 每 await 都 checkpoint flow run 级别记录 无 · 脚本死了就死了
跨进程 resume worker 换了能续 flow 重启即重跑 无概念
等待外部事件 aio_wait_for_event
(CEL filter, 跨重启)		 不支持
(只能 poll)		 不支持
定时触发 有 schedule，但少用 deployment 自带 cron 调度 crontab 原生
worker 池横向扩展 支持但单 worker 够用 主打能力 (work pool) 单机一份
失败重试 max_retries 参数 retry policy 脚本自己写
观测 UI Hatchet 后台 Prefect Cloud / Server 看 stdout / 日志文件
部署成本 需要 Hatchet server 需要 Prefect server + worker 零额外组件
关键观察： cron 在很多维度都"输"，但它的强项是不依赖任何外部组件 —— Hatchet 挂了它照跑、Prefect server 重启它不知道。这种"调度独立性"在"必须送达"的报告类任务上反而是优势。

3Hatchet 这一层 · 为什么 agent 业务流非它不可

Hatchet 不是普通后台 task，是"durable workflow engine"。它的核心承诺：workflow 状态由 server 端持有，worker 只是"执行肌肉"，可以随时换。在 agent 场景里这个承诺有几个具体含义：

execution_timeout = 60m
一个 task 可以名正言顺跑 60 分钟。LLM 慢 + 工具调用 + 等人都算在内，不被中间件杀掉。
DurableContext
不是普通 Python context —— 每次 await 时把当前 frame 状态序列化到 server。worker 换了能在原 await 处恢复。
aio_wait_for_event
workflow 主动"挂起等事件"。worker 完全空闲，不像普通 asyncio coroutine 那样占内存。
固定 event type + CEL filter
"踩坑笔记"：不能用动态 event key（重启后 server 没注册会丢消息）。固定 type 在 server 端永远 "known"，CEL 服务端筛 run_id。

把"用户点 IM 按钮"建模成"workflow 等一个 event"，是 Hatchet 这层最值钱的能力。换成 Prefect 或 cron 都做不到 —— 它们没有"挂起等远程事件"这个原语。

判断准则： 任务里只要有"等人/等回调，可能等几小时甚至跨天"这种环节，就必须放 Hatchet。这是它唯一能干别人干不了的事。

4Prefect 这一层 · pipeline 的"工厂车间"

Prefect 的强项是把"批量数据加工"切成 flow → tasks，分发到 worker pool 上跑，带观测和重试。具体形态：

组件 角色 类比
flow一个 pipeline 的入口函数"流程图本身"
taskflow 里的一个步骤"流程图里一个方块"
deploymentflow 在哪个 worker pool 上、什么时间跑"派工单 + 排班表"
work pool一组 worker 共享一个名字，flow 进池子分发"车间"
worker实际拉起 flow 跑的进程"车间里的工人"

在 agent 平台里，Prefect 主要承担：

达人/视频采集
夜里跑外部爬虫 → 入库到内部 creator pool。一次几万条，分批 task。
video_index DAG
视频内容理解 pipeline，对接 OCR / 转写 / 标签生成。失败重试在 Prefect 层。
reflection_agent
把 agent 跑过的会话做反思总结、写回 memory。批量离线跑。
日报聚合
汇总各 agent 的 token 用量、cost、调用次数，生成 IM 卡片。
判断准则： 任务"不依赖人交互，但要批量跑 + 横向扩 + 有观测"，进 Prefect。它的 work pool 模型让"加机器就能加吞吐"成立。

5cron 这一层 · 看似最土，但最不会挂

cron 是宿主机自己的 crontab 加一个 Python 脚本。没有 server、没有 worker、没有 UI。看起来很原始，但有一类任务它最合适：

✓ cron 适合什么
  • 每天定时发一份必须送达的报告（花费、活跃、健康检查）
  • 任务逻辑极简单 —— 几十行 Python，没分支
  • 调度独立性要求高 —— Hatchet/Prefect 都挂了它也要送达
  • 跑在容器/服务的同一台 EC2，能直接读容器环境
  • 失败了发个 IM 告警就够，不需要复杂的重试
✗ cron 不适合什么
  • 任务要跑 > 10 分钟（cron 没好用的超时控制）
  • 逻辑要分支 + 等人 + 多 step
  • 需要观测 UI / 历史回放
  • 需要横向扩到多机
  • 失败了要自动重试 N 次

实际部署形态：脚本放在 EC2 的 /opt/<product>/scripts/ 下，由仓库的 GitHub Action 自动 rsync 同步过去。crontab 写一行 0 9 * * * python /opt/.../daily_report.py。典型代表：每日 API cost 报告、token 用量汇总、容器健康检查。

容易踩坑： 新加 cron 脚本时，必须同步登记到 CI 部署 workflow 的白名单，否则下次 deploy 不会同步过去。这条很容易忘 —— 因为 cron 没有"中心化部署"概念，全靠仓库 → 宿主的 sync。

6一个任务该放哪层 · 决策树

给一个新任务做层选择，按下面顺序问 3 个问题就够：

Q1. 任务里有"等人交互/等远程回调，可能挂起几小时甚至跨天"吗？
  ├─ 是 →  Hatchet 没得选
  └─ 否 → 继续问 Q2

Q2. 任务需要"批量分发、横向扩 worker、观测后台、失败重试"吗？
  ├─ 是 →  Prefect 主场
  └─ 否 → 继续问 Q3

Q3. 任务是"每天定时发个简单报告 / 跑个简单脚本"，且必须送达？
  ├─ 是 →  cron 最稳
  └─ 否 → 重新看任务 spec，可能是错的问题
反例提醒： "好像 Prefect 也能做日报" —— 能做，但带来一个隐患：Prefect server 挂了或 worker pool 路由错了，日报就没了。日报这类"必须每天送达"的任务，挂在 cron 上比挂在 Prefect 上少一层依赖、少一种失败模式。下面第 7 节就是一个真实事故。

7真实事故 · 路由错 worker，日报全丢

把决策树落到一个真实案例上 —— 一个"看起来 Prefect 合适、实则掉链子"的故事：

事故 · skill-usage-daily-report 静默丢失

一份每日技能使用报告，连续几天没到 IM 群

表象
"昨天的日报没收到 · 没报错 · Prefect UI 显示 run success"
环境
有两个 Prefect work pool：ec2-worker-v2（自家 EC2）和 my-ec2-pool（云上托管）
现象
近期 daily-report 的 flow run 100% 落在云上 worker；云 worker 发 IM interactive-card 100% 失败；EC2 worker 100% 成功
根因
云上残留一个旧 ProcessWorker，它能拉到任务、能跑、但所在网络出口 POST IM 卡片端点失败 —— 而 flow 代码把 IM POST 失败吞掉只 log warning，Prefect 角度看 "task succeeded"
为何静默
Prefect 失败重试只保护"task 抛异常"，对"业务上失败但返回 success" 无能为力
启示一： Prefect 的"成功"是 task 函数没抛异常，不是业务真送达。
启示二： 多个 worker pool 共用一个 flow，路由是隐藏失败面 —— 同一份代码在不同 pool 上行为不一样，UI 看不出来。
启示三： "必须送达"的报告，挂 cron 反而少一层路由不确定性。如果非 Prefect 不可，也要在 flow 末尾做"送达验证 + 失败 raise"，否则它会骗你。

8三层协作 · 一个真实场景的全链路

看一次"早 9 点的 campaign 复盘"是怎么三层接力跑完的：

08:55
cron · 触发拉取前置数据
EC2 crontab 跑一个 bootstrap 脚本：检查昨日 campaign 是否已结算 → 写一个 marker 文件
09:00
Prefect · 批量入库
deployment 触发达人数据 flow，10 个 task 并发拉外部接口，入 creator pool 表，失败重试 3 次
09:08
Prefect · 视频索引补齐
video_index flow 把昨天 campaign 涉及视频做转写 + 打标，结果写回 metadata 表
09:15
Hatchet · Athena autopilot 启动
campaign-autopilot durable task 启动，开始跑 7 步流程，前两步是数据汇总
09:35
Step 3 · 等运营点确认
underperformer 名单生成 → 发 IM 卡片 → workflow 调 aio_wait_for_event 挂起
10:00
容器部署新版本
worker 被替换。Hatchet server 不动，workflow 状态安全。worker B 在 10:01 接管
11:20
运营点 confirm
IM button callback → POST event 到 Hatchet → CEL filter 匹配 run_id → workflow resume Step 4
11:35
Step 4-7 顺序跑完
budget 调整 + reflection 写回 memory + 生成 Daily Report 卡片推送
18:00
cron · 当日花费报告
EC2 上独立脚本聚合当日所有 agent 的 API cost，推送到老板群 —— 跟前面 Prefect/Hatchet 一切无关
仔细看时间线： 三层不是上下调用，是各自管自己性格匹配的环节 —— Prefect 管"批量数据备好"、Hatchet 管"agent 主流程 + 等人"、cron 管"准点送达的简报"。哪层挂了，其他两层照常工作 —— 这就是分层的工程价值。

最后一句话： 调度器不是越强越好。每多用一个"全能调度器"做它不擅长的事，就多一个静默失败的可能。三层并存的代价是工程师要懂三套东西，但收益是 —— 每个任务都跑在自己性格匹配的肌肉上，挂一层的时候其他两层还在跑。
