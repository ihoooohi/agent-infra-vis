11 · 为什么线上和代码不一样

Memory 系统演化考古 · 从 git log 重建 5 次重写背后的逻辑
第 02 篇发现 Athena 的 memory 系统跟代码里的 MemoryStore/Consolidator/Dream 几乎不重叠 —— 6/10 项 dead code。这不是偶然。
Git log 显示这个模块在 17 个月内被改过 18 次，至少经历了 5 次重大重写。每次重写都是被具体的线上问题逼出来的。这篇文章把 18 个 PR 重新排队，看清楚分歧的因果链。

0TL;DR · 三层原因

"线上和代码不一样"有三层原因，从浅到深：

SURFACE
表层 · 技术不兼容
13 agent 共享 workspace 让 GitStore 自动 commit 必会冲突；jsonl 多进程 append 会乱；Dream cron 跑 LLM 太贵。框架的优雅设计在 Athena 的部署形态下根本跑不起来。
MIDDLE
中层 · 产品 vs 框架愿景分歧
nanobot-ai 设计的是"个人 AI 助理"（一个用户、一个 workspace、自动归档）。Athena 是"团队 AI 经理"（13 agent、共享知识库、人工把关）。两者的 memory 需求从根上就不同。
DEEP
深层 · fork 后的必然分叉
PR #1906 "fork nanobot-ai + fix race"。fork 那一刻，Athena 和上游 nanobot 就走上了两条路。之后 18 个 memory PR 全是 the company 自己加的，没有任何一个回上游。两年下来已经是两个项目。

25 个具体决策 · 都是被问题逼出来的

从 PR 里提取 5 个关键技术决策。每个都有具体问题 → 具体解决的因果，不是凭空设计：

#1
放弃 GitStore 自动 commit
13 个 nanobot 容器挂载同一份 workspace，并发 commit MEMORY/SOUL/USER 时 git 会 lock 冲突 / detached HEAD / 时序错乱。
workspace 不做 git 跟踪，改用 .md 文件 + AGENTS.md 协议（人工约束哪个 agent 写哪一段）。
workspace 没有 .git；PR #1717 "memory isolation DM/Group" 时期发现的
#2
不用 history.jsonl shard
jsonl 多进程 append 没有强一致性，13 个 agent 同时 append 会出现行截断 / 互覆盖。Consolidator 设计假设单进程。
按日 markdown 文件 memory/YYYY-MM-DD.md，每个 agent 自己用 edit_file 工具按 section 写自己负责区。
实测 100+ daily .md 文件 · 没有 history.jsonl · PR #1744 "workspace-first"
#3
Dream 离线整理 → 撤掉
Dream 每次跑都调 LLM 摘要 + 编辑文件。13 agent 各自 cron 跑成本 13×。还经常发生"幻觉删除"重要内容 → 人工恢复。
不再有自动 Dream cron。MEMORY.md 维护改成人工 + agent 在对话中 edit_file 半自动。需要时再人工触发"compress"。
.dream_cursor 4/23 后 20 天没动 · MEMORY.md "Last compressed: 2026-02-14" 是人工标记
#4
Consolidator 实时归档 → 撤掉，改 5 阶段 hook
Consolidator 在 token 紧时同步调 LLM 摘要 → 用户响应慢一倍。而且写到 history.jsonl 又没人读（见决策 #2）。
hooks/compaction.py 5 阶段 progressive pipeline：60% offload(文件) → 70% prune(字符串 dedup) → 70% truncate(JSON 截断) → 85% DeepSeek INDEX → 92% emergency。前 3 阶段零 LLM 成本。
PR #1612 引入 → #1910 progressive disclosure → #1986 align OpenClaw → #2186 aggressive dump
#5
pgvector 从"自动注入"改"工具调用"
把 memory_search 结果默认塞 system prompt → 每次都消耗 token，agent 也未必需要。而且 chat scope 隔离逻辑很难放到 system prompt 阶段。
改为 active memory_search 工具，agent 自己判断何时调用。chat_scope 是工具参数。
PR #2961 (P0 follow-up to #2741) "active memory_search tool with chat scope"

3设计哲学 · 两种愿景的对撞

表层技术问题背后是两种"agent memory 应该长啥样"的根本性分歧：

FRAMEWORK · nanobot-ai 原作者愿景
个人 AI 助理 · 自动管理
"一个用户 + 一个 agent + 一个 workspace · agent 自己悄悄地把所有事情打理好"
  • workspace 私有（git 跟踪自己改的 SOUL.md）
  • 对话历史都自动归档 (jsonl shard)
  • token 紧 → Consolidator 静默摘要（用户不察觉）
  • 每天 Dream 离线整理（不打扰用户）
  • memory自动注入 system prompt
  • "agent 越聪明，用户越省心"
LIVE · Athena 团队的实际选择
团队 AI 经理 · 显式协作
"13 agent + 多个用户 + 共享知识库 · agent 是团队同事，要遵守协议、写 daily log、被人审核"
  • workspace 共享（多人多 agent 协作，不能私有）
  • 对话不归档，每天写 daily log .md（人也能读）
  • token 紧 → 纯字符串压缩，必要时 DeepSeek INDEX
  • 不自动整理，人工触发"compress"（高质量胜过频率）
  • memory 是工具 · agent 自己决定何时调
  • "agent 越守规矩，团队越能用上"
两种愿景都对，但不能同时存在。 框架愿景适合"个人副驾"场景（Cursor / Claude Code 那种）。Athena 愿景适合"团队代理"场景（slack/lark bot 给一队人用）。Athena 选择的是后者，所以前者的代码必然变 dead。

4fork 演化 · 不可避免的分叉

PR #1906 "fork nanobot-ai + fix _sent_in_turn race condition" 是个转折点 —— the company 不再依赖上游 nanobot-ai，开始维护自己的 fork。这是命运分叉时刻：

"fork 后零回流"是个 social signal：the company 没把这些改动 PR 回上游 nanobot-ai。可能原因 ——
  • 改动太 Athena-specific（共享 workspace / 13 agent / Hatchet 集成），上游用不到
  • 上游对 PR review 要求高，the company 没时间走流程
  • 核心改动违反上游设计哲学，合不了
任何一个原因都会让 fork 不可逆。

5三层真相 · 由浅入深

🔧
浅层 · "代码经常死，因为环境变了"
框架代码是当时(2024 Q4)对"一个 agent / 一个 workspace"的设计。后来 Athena 变成 13 agent 共享 + Hatchet durable task + 多用户 IM 群 —— 同一份代码在新场景下不能工作，要么改要么弃。

线上不一样不是 bug，是代码没跟上场景的自然结果。
⚖️
中层 · "通用框架做不出专用产品"
nanobot-ai 框架想做的是"generic AI agent framework" —— 像 LangChain。Athena 想做的是具体的产品 —— 团队 IM platform 机器人。

通用框架追求覆盖所有可能场景（所以有 Consolidator、Dream、GitStore 等优雅抽象）。产品追求解决具体问题（所以选了朴素的 markdown + workspace 共享）。

两个目标不可调和 → Athena 必然选自己写一份。
🌳
深层 · "fork 是分叉树的起点，不是修复"
很多团队误以为"fork = 临时方案，等改完再回去"。实际上 fork 之后有自己的演化压力 —— bug 要修、feature 要加、deadline 要赶 —— 没人有功夫保持兼容上游。

the company 在 PR #1906 之后再没回过上游。所有 17 个 memory PR 都是单方面改 fork。两年下来 Athena 用的"nanobot"已经是事实上的独立项目，只是没改名。

类似的故事在 Linux fork（Android）、MongoDB fork（FerretDB）、ElasticSearch fork（OpenSearch）反复发生 —— fork 是命运分叉，不可逆。

6实用启示

看代码要看 git log
单读源码 = 看到"所有可能存在过的设计"。读 git log = 看到"哪些活下来了"。dead code 在 git log 里有特征：长期不再被改、附近 PR 都绕开它。
cursor / state 文件是化石
.cursor / .checkpoints / .progress 这种 runtime state 文件，mtime 比代码 mtime 重要 —— 文件几周不动 = 对应模块没在跑。
PR 名暗藏版本数字
"memory-v7" / "P1+P2+P3" / "overhaul" / "rework" 这种关键词 = 这块被改过很多次。新人看到这种 PR 名要立刻警觉"我读的这版可能也不是最终版"。
fork 不要轻易做
the company fork nanobot-ai 后写了 17 个 PR 全靠自己 —— 这是巨大维护成本。fork 前先问"能不能 upstream"，不行再开 fork。
dead code 要主动标"deprecated"
nanobot/agent/memory.py 里 Consolidator/Dream 还在但没人用。建议加 __deprecated__ = "Use hooks/compaction.py instead, since 2025-04" 等显式标记，下次同事再读就不会被误导。
CLAUDE.md 要同步更新
CLAUDE.md 还在写"Memory Service uses VectorDB + Gemini ..."但 compaction 5 阶段一字不提 —— 文档落后实际 5+ 个 PR。让团队定期审计 onboarding 文档跟代码的一致性。

7一句话总结

线上和代码不一样 ≠ bug。这是 fork 后两年自然演化的结果。
代码反映"当年想做什么"，线上反映"最后做出了什么"。
中间是 18 个 PR、5 次大重写、1 次 fork、2 年时间，所有这些都没有义务保持原始设计一致。
要理解 Athena，必须把代码当地质剖面读 —— 不同地层是不同时期的产物，活的层在最上面，下面是化石。
