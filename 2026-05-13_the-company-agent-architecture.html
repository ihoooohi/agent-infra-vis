the company Agent 平台 · 功能模块图

从"系统在做什么"出发，看清楚每个功能模块的位置和关系
the company 把 AI 塞进 IM 群，让它像团队成员一样工作 —— 你 @ 它一句话，它能查数据、写报告、回邮件、调内部系统。

5双容器协作 · gateway 与 nanobot

前面讲的"一个 Bot"在实际部署里其实由两个容器配合：gateway 管"门"，nanobot 管"脑"。一个接消息分发，一个负责思考和执行。
接收 / 路由
gateway
站在 IM platform 入口处的"门面"
  • 接收 IM webhook 推送
  • 校验签名、过滤重复消息
  • 识别消息属于哪个 bot / 租户
  • 把请求转给对应的 nanobot
  • 把 nanobot 的回复发回 IM platform
⇆
内网 HTTP
思考 / 执行
nanobot
真正"动脑"的部分
  • 加载这个 bot 的人格 prompt
  • 读取长期记忆和对话历史
  • 调用 LLM 推理
  • 调工具拿数据、跑分析
  • 组装答案，让 gateway 送出去

关注点分离

gateway 只做"门面"工作（协议、签名、路由），nanobot 专心"业务"（思考、记忆、工具）。两边代码不打架。

故障隔离

nanobot 升级 / 慢 / 崩了，gateway 还能继续接消息、排队、回降级提示，不会丢请求。

独立扩展

gateway 轻量好水平扩；nanobot 重（要持有记忆、长上下文），独立部署独立调资源。

选最顺手的工具

gateway 用 TypeScript（IM platform SDK / webhook 生态成熟），nanobot 用 Python（LLM / 工具调用生态成熟）。

Athena 模式 · 共享 gateway → per-bot nanobot
所有内部 bot 共用一组 gateway，再分发到自己专属的 nanobot。bot 之间像住在同一栋楼的同事 —— 公共大门一个，每人有自己的工位。

6关键功能特性

同大脑、不同人格
所有 bot 用同一套底层引擎，靠 人格 prompt + 工具集 + 记忆 这三件套捏出不同的角色 —— Athena 像营销助理、Echo 像数据分析师，骨架一样，外壳完全不同。
持久记忆
每个 bot / 租户都有自己的长期记忆，跨对话保留上下文 —— 上次说过的偏好、确认过的结论、对接过的人，下次还在。
工具增强
Bot 不只是聊天，能 查内部数据库、调外部 API、读写文件、跑分析脚本。LLM 自己决定什么时候用哪个工具，可以连续调用直到拿到答案。
租户级隔离（Cetus）
外部客户最关心数据安全。Cetus 把每个租户的 prompt、记忆、API 配额、对接的工具，全部物理 + 逻辑双重分开 —— 一家的故障不会影响别家。
