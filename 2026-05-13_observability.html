10 · 监控与可观测性

OTEL + Jaeger + 一套有创意的"agent 自我承认丢上下文"检测
Athena 的观测分三层：分布式 trace（OTEL → Jaeger，每个 turn 一个 span 树）· 结构化 metric（context_metrics.jsonl，5 种事件 append-only）· cost / usage（UsageMeter → IM platform 日报）。
最有创意的是 ctx_loss 检测 —— 用正则在 agent 自己的输出里抓"我这边没看到"等措辞，把"上下文丢失"变成可量化指标。Hand-verified on 17 hits across 943 historical pairs.

2OTEL · graceful no-op

OpenTelemetry 集成最讨厌的事 = 没配 endpoint 时崩 / 拖慢。otel_setup.py 用 "可选启用" 模式：

def init_otel(service_name: str) -> bool:
    endpoint = os.environ.get("OTEL_EXPORTER_OTLP_ENDPOINT")
    if not endpoint:
        logger.info("OTEL tracing disabled")
        return False   # 直接退，零开销

    try:
        provider = TracerProvider(
            resource=Resource.create({"service.name": service_name})
        )
        exporter = OTLPSpanExporter(endpoint=endpoint, insecure=True)
        provider.add_span_processor(BatchSpanProcessor(
            exporter,
            max_queue_size=2048,
            max_export_batch_size=256,
            schedule_delay_millis=5000,
        ))
        trace.set_tracer_provider(provider)
        return True
    except Exception as e:
        logger.warning(f"OTEL init failed: {e}")
        return False

关键设计：

  • env 不设 = 完全 no-op，没 import 开销
  • init 失败不抛错，agent 照常工作
  • OtelHook 同样检查 _OTEL_AVAILABLE，所有 hook 方法变 no-op
  • BatchSpanProcessor：5s 批量发送，2048 缓冲，避免每个 span 一次 RPC

3Span 树结构

每次 agent turn 产生这样一颗 span 树（来自 OtelHook 注释）：

▼ agent-turn (real duration)
├─ think-iteration-0 (real duration)
│    ├─ tool:exec (marker, zero duration)
│    └─ tool:read_file (marker, zero duration)
└─ think-iteration-1 (real duration)
     └─ tool:web_fetch (marker, zero duration)
Phase 1 妥协： tool spans 是 zero-duration markers —— 只记 name + 完成 status，没记每个 tool 实际耗时。原因：要改 nanobot 框架的 runner.py 加 timing hook。Phase 2 计划把这个补上。
Phase 1 已经能回答"哪个 turn 调了哪些工具"，但不能回答"web_fetch 花了多久"。

每个 span 都带这些 base attribute：

service.name = "agent-primary"   # 哪个 agent
session.key  = "lark:oc_abc..."
chat.id      = "oc_abc..."
trace.id     = "<run uuid>"     # 与 _active_cards trace_id 一致

所以在 Jaeger 里能用 chat.id 或 trace.id 反查到具体某条 IM platform 消息的完整链路。

5context_metrics.jsonl · 5 种事件

OTEL 适合"看单个 trace"，但统计聚合更适合 JSONL。{workspace}/context_metrics.jsonl append-only，每行一个事件：

turn
每次 LLM 调用后写一条
prompt_tokens · completion_tokens · cached_tokens · context_window · used_tools · model · ttft_ms · chat_type · session_age_hours · message_count
snip
_snip_history() 删 msg（context 溢出兜底）
removed_messages · saved_tokens · reason
compaction
Consolidator 归档老消息到 history.jsonl
boundary_idx · archived_tokens · session_key
session_reset
session 被重置 — 按原因分类计数
reason · session_age
ctx_loss
agent 回复匹配了"丢上下文"的高精度模式
pattern_name · session · chat_id · reply_excerpt

一条 turn 事件样例：

{"event":"turn", "ts":"2026-05-13T08:42:11Z", "session":"lark:oc_abc", "iteration":2, "prompt_tokens":12480, "completion_tokens":450, "cached_tokens":11200, "context_window":618000, "used_tools":true, "model":"...claude-opus-4-6-v1", "ttft_ms":823, "chat_type":"group", "session_age_hours":3.4, "message_count":42}
append-only JSONL 而非 DB： 一台 nanobot 容器一天约 10K 条事件 ≈ 几 MB JSONL，没必要 DB。Prefect 每天聚合这些文件成 IM platform 日报。本地查也方便 (jq 一把梭)。

6ctx_loss 检测 · 让 agent 自己暴露问题

传统 "context loss" 很难测 —— 你怎么知道 agent 是"真不记得"还是"故意不答"？ Athena 团队的招：正则匹配 agent 自己说漏的话。

3 个高精度模式（hand-verified 17 hits / 943 历史样本，Apr 2026 sweep）：
pattern_name
匹配什么 + 例子
agent_no_context
agent 承认看不到/找不到 prior 消息/对话/历史。故意排除"don't have access to repo / HR DB"（domain knowledge gap，不是 memory 失败）
我这边没看到刚才那条消息 · don't have visibility into your earlier reply
agent_request_resend
agent 要求用户重发。注意：必须带 "again / 再" 才算 —— "Could you share the doc?" 是初次请求，不是 memory 失败
你能不能再发一遍 · could you resend that · please re-send
agent_no_history
agent 明确说 "没找到 / 看不到 历史 / 聊天记录"
没有找到聊天记录 · can't find that message

检测时机（在 _LoopHook.after_iteration 里）：

# 只检查 final user-facing 回复（不再有 tool_calls）
# 中间 "let me check..." 不算 ctx_loss
if not context.tool_calls:
    final_text = getattr(context, "final_content", None)
    if final_text:
        tracker.record_ctx_loss(
            reply_text=final_text,
            session=self._session_key,
            chat_id=chat_id,
        )

核心设计原则：

  • 高精度优先于召回 · 宁可漏，不能误报。误报会让指标失真
  • 只看 final reply · "我来查一下"（中间状态）不算丢失
  • 人工 verify 阈值 · 17 hits / 943 pairs ≈ 1.8% ctx_loss rate
  • 迭代精度 · pattern 注释提到 "Codex round 2 / round 3" —— 用 LLM 对 pattern 做对抗式 review
这个 metric 给团队的价值： 每天 IM platform 日报里能看 "ctx_loss rate = 1.7%"。如果某天突然 5%，说明context 处理坏了（compaction 太激进 / memory.search 挂了 / 新部署 regression）—— 主动 alert 而不是等用户抱怨。

7TTFT · 流式响应感知

用户感知的不是总时长，是第一个字什么时候出现（Time To First Token）。AgentLoop 的 _LoopHook 在每次 iteration 测它：

async def before_iteration(self, context):
    self._turn_start = time.perf_counter()
    self._ttft_ms = None

async def on_stream(self, context, delta):
    if self._ttft_ms is None and self._turn_start > 0:
        self._ttft_ms = (time.perf_counter() - self._turn_start) * 1000
    # ... 转发 delta 给 callback

第一个 stream chunk 到达时记 ms 数。每次 turn 事件 JSONL 写 ttft_ms，统计后能看：

  • 不同模型的 TTFT 分布（Opus vs Sonnet）
  • 不同群聊 vs 私聊的 TTFT（context 大小不同）
  • 不同session age 的 TTFT（老对话 context 大、TTFT 高）

8UsageMeter · 钱花在哪

每次 LLM / embedding 调用都过 UsageMeter 记账（第 4 篇详谈过路由 + 计费）。关键字段：

tokens_input / tokens_output
基础计费量，按模型单价算钱
cache_read_input_tokens
Anthropic prompt cache 命中部分 · 价格 0.1×
cache_creation_input_tokens
第一次创建 cache · 价格 1.25×（贵 25%）
fallback_context
走 fallback 链时记到哪一级 · 分开统计各路成本
IM platform 日报来源： 这些数据每天被 /opt/Athena/scripts/amy_cost_daily_report.py 聚合，推到专属 IM webhook。团队能看到"今天 Sonnet vs Opus vs Nova Micro 各花多少"。这正是难度路由（第 4 篇）的反馈环。

10观测的"哲学" · 信号 → 改进

Athena 的观测设计有几个一致的原则，值得总结：

  1. 没有 endpoint 配置就 0 开销 · 不让观测拖累生产
  2. 异常隔离 · 观测代码挂了不能挂 agent —— AgentHook 全方法 catch + log
  3. 原始数据 append-only · JSONL 比 DB 便宜，聚合在外
  4. 指标驱动决策 · 看到 ctx_loss 涨 → 改 compaction · 看到某模型成本占比异常 → 调整路由阈值
  5. 人工 verify 模式 · 不发 "随手加的 metric"，正则 / 阈值都要人工对照样本
  6. trace_id 全链路 · 同一个 UUID 串起 OTEL span / JSONL 事件 / IM 卡片 / Hatchet workflow run，反查任意一处都能找到全貌
