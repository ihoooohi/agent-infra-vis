PART 04

改造路径 · 决策树

按"成本 / 收益 / Athena 形态是否真需要"分档。
从 audit 结论到改造方案
三层决策：N/A 砍 → ✓ 仅监控 → ≈/✗ 分 S/M/L 三档。S 档全是"代码已存在 / 仅基础设施替换"，不是真新建。
S1 ★★★
⑩ 接通 Dream system job
≈ 50-80 行 · 0.5 工时
把当前 CLI 启动路径里做的定时任务注册（约 30 行）搬到 fleet 容器启动流程，加 leader 选举（自演化器只在主 bot 跑）。代码已存在，只接连线。
~ fleet 容器启动文件 ~ fleet 配置 + leader 选举测试
S2 ★★★
⑮ Vault → AWS Secrets Manager
≈ 80-120 行 · 1 工时
把容器配置里的 API key 抽到 AWS Secrets Manager，启动时拉。Athena 这种 fleet 部署应该早就做——key 在 .env 跟随 EC2 实例同生死。
+ 秘密加载模块 ~ 容器启动流程 ~ 部署编排
S3 ★★
⑱ 加 audit_events 表 (按身份)
≈ 60 行 · 0.5 工时
向量库加一张审计事件表（actor / action / resource / ts），在记忆检索 / 文件写入 / IM 发消息三个关键工具里埋点。主要是给"出事可追溯"兜底。
+ 审计表 schema ~ 记忆检索工具 ~ IM 发消息工具
M1
⑪ VectorDB 启用真 RLS
≈ 100 行 · 1 工时
trigger：开始有非 The Company 团队进来。托管向量库底层是 Postgres，可以启用真行级安全策略。当前 Python 后置过滤可保留作 belt-and-braces。
+ RLS 策略 schema ~ 记忆模块身份注入
M2
⑦ 用量记账加限流
≈ 150 行 · 1-2 工时
trigger：单 agent 日成本超阈值。按 agent/day 加 token 桶，超限切便宜模型 / 拒绝新会话 / 报警。
~ 用量记账模块 + 限流模块
M3
⑯ 加 Org 层 SOUL
≈ 200-300 行 · 1-2 工时
trigger：业务合规要求 / 外部租户。新增组织级 SOUL（合规 guardrail），按组织 → 团队 → 皮肤顺序拼接系统提示。
+ 组织级 SOUL ~ 上下文拼接逻辑
L ✗
⑫ 独立 Meilisearch
不建议
Athena workspace 量级（百级 daily shard）ripgrep 几十毫秒搞定。独立检索引擎要部署集群 + CDC 同步，YAGNI。等 shard 量级到 10k+ 再说。
不动
L ✗
⑬ S3 分层归档
不建议
Athena 单 EC2 fleet · workspace 块存储共享挂载 跑得稳。S3 lifecycle (30d→90d→Glacier) 只在合规留存要求强 / 容量上 TB 时才划算。
不动
L ✗
⑤⑥⑭ OIDC / Tenant Resolver / Redis
形态决定不做
Athena 是单组织内部 fleet，不接外部用户，没多租户切换需求。Redis 在多 host 部署才有价值——Athena 单 EC2 用 asyncio.Lock 已经够。
不动
PART 05

风险 + 执行时间轴

3 个 S 档建议合并到 1-2 个 PR 里做。
! 别强行做 N/A 件
原文花大篇幅讲 OIDC / RLS / Redis，Athena 形态用不上。看到"团队版必备"就照搬是误读——它的"团队版"是 SaaS 多租户，Athena 是单组织 fleet。
! Vault 改造分步走
.env → Secrets Manager 不要一次性切。先把新 key 写入 SSM，旧 key 保留 .env 兜底，切换 N 周后再撤 .env。CLAUDE.md 有踩过 .env 含空格的坑 (PR #2587)。
! Dream 接通的并发约束
13 容器共享 workspace。Dream 必须 leader 选举（只 athena 跑），否则 Phase 2 LLM 增量改 SOUL.md 会并发覆盖。前一篇查过细节。
! 审计日志成本
⑱ S3 加 audit_events 表，只埋"安全敏感动作"（memory_search / file_write / send_message），不要给所有工具埋——会涨 VectorDB 成本。
推荐执行节奏 · 时间轴
第一个 PR 把 S1+S2+S3 合并 (≈ 2 工时)，M 档全部业务触发再说。N/A 三件永不做。
整篇结论：用原文给的 18 件团队助理标尺审视 Athena，分布是 ✓5 / ≈8 / ✗2 / N/A 3。3 件 N/A 是 Athena 形态决定不该做（OIDC / Tenant Resolver / Redis），5 件已达标（接入层 4 + 用量记账 + 计算层 2），剩下 10 件里只有 3 个 S 档值得立刻做：⑩ 接通 Dream cron · ⑮ 改 AWS Secrets Manager · ⑱ 加 audit_events 表。M 档 3 件（RLS / throttle / Org SOUL）等业务触发。整体改造工作量 ≈ 2-5 工时。误读原文最大的风险是"看到'团队版必备'就照搬"——文章里的团队版是 SaaS 多租户，Athena 是单组织 fleet，两者不是同一个产品形态。
