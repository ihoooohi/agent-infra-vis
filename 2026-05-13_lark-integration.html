05 · IM platform 接入 + 卡片生命周期

用户感觉 Athena "正在打字" 是怎么造出来的 —— 卡片状态机 + 实时更新
IM platform 没有 "AI 正在打字..." 协议，所以 Athena 自己造了一套：发出去一张 IM platform 卡片当占位，然后不停往里塞内容 —— "我在思考" → "我在调 web_search('XXX')" → "我在写文件" → 最终答案。整个过程靠 trace_id → card_id 这个内存映射追踪每张活动卡片。

2卡片状态机 · 5 个状态

用户发消息 → 立即收到一张"占位卡片" → 卡片内容随 agent 工作进度原地更新。这就是 Athena 给人的"实时打字感"。每个状态对应 cards.py 里一个 build_* 函数：

thinking
刚收到消息
"我在思考你的问题..."
working
LLM 在调工具
"我在用 web_search 查 ..."
final
答案出来了
展示完整内容 + tool 历史
stopped
用户按了 stop 按钮
中止当前轮
error
LLM 调用挂了
展示错误 + trace 链接

每张卡片底部固定带：model 名 / 难度 tag / Stop 按钮 / trace 链接 / 排队数。这是 cards.py 各 build_* 函数的共同参数。

4追踪机制 · trace_id → card_id 内存映射

每次有 agent 任务在跑，server 内存里维护一个 _active_cards 字典：

# nanobot_server.py · line 257-262
_active_lock = threading.Lock()
_active_cards: dict[str, dict] = {}    # trace_id → {card_id, started_at}
_active_reactions: dict[str, tuple[str, str]] = {}  # trace_id → (msg_id, reaction_id)
_ACTIVE_MAX = 10_000  # 上限，超过驱逐最老的

用 threading.Lock（不是 asyncio.Lock）—— 因为 FastAPI webhook 跑在 asyncio loop，Hatchet workflow handler 跑在独立线程，两边都改这个 dict 必须 thread-safe。

5设计反转 · 从持久化回到纯内存

CLAUDE.md 上写"card_lifecycle.py — SQLite store · 30s background reconciler 修复孤儿卡片"。但代码实测：这个文件不存在了。被有意删除：

# nanobot_server.py · line 87 # card_lifecycle removed — in-memory tracking + shutdown hook replaces it. # See .respec/nanobot-card-lifecycle-redesign.md for rationale. # nanobot_server.py · line 5390 # One-time cleanup: delete legacy card_lifecycle.db from previous deploys. _legacy_db = Path("/workspace/sessions/card_lifecycle.db") ...

原因写在代码注释里：

为什么删 SQLite： "OpenClaw-IM platform + Hermes 都只在内存里 hold outbound IDs。持久化反而造成 zombie row（shadow-noop 状态留下死记录），错误地泄露进 IM platform API 调用。"

简而言之：SQLite 持久化导致死信息复活，比"重启丢卡片"还糟。

新方案 = 纯内存 + shutdown hook 善后：

  1. 正常运行时：所有 active 卡片只在 _active_cards 内存里
  2. 容器 shutdown：FastAPI @app.on_event("shutdown") 触发，遍历 _snapshot_active_cards()
  3. 对每张活动卡片发"interrupted"终态更新："这次任务被中断了，你可以重新问"
  4. 容器停掉，所有内存清空 —— 重启后是干净状态
代价： 容器异常崩溃（没走 shutdown hook）会留下"幽灵卡片"—— 用户那边永远停在 thinking 状态。但这种事件极少（Hatchet 一般正常 graceful），用户重发消息也能恢复 —— 团队认定比 SQLite zombie 风险小得多。

6ToolTracker · 边跑边显示"我在做什么"

working 卡片那个"我在调 web_search('XXX')"是 lark/tracker.py 里的 ToolTracker 状态机渲染出来的。它持有一个 list[ToolCall]，每次有新 tool 调用 / tool 完成就更新：

class ToolCall:
    name: str          # "web_search"
    family: str        # "web" / "filesystem" / "shell" / ...
    target: str        # "tenant-01 Q2 metrics" / "/path/file.py"
    started_at: float
    finished_at: float | None

class ToolTracker:
    def add_call(...)
    def complete(...)
    def render() -> str     # 给卡片 body 用

渲染时 ToolTracker 做几件事：

  • _icon_for(name) · 不同工具给不同 emoji 图标 —— 🔍 web_search · 📝 write_file · 🐚 exec · ...
  • _get_family(name) · 把工具归类（fs / web / shell）—— 同 family 的连续调用可以折叠成一行显示
  • _extract_target(detail, family) · 从 tool args 抽出最关键参数（"web_search(query='X')" → "X"）
  • _parse_single_hint(hint) · 把 agent 写的中文活动描述渲染成 inline icon

7Emoji Reaction · 第二条反馈通道

除了卡片，Athena 还在用户的原消息上加 emoji reaction（"👀 我看到了" / "✅ 处理完了"）。这是用户最快收到 "Athena 知道你了"的信号 —— 因为发卡片本身可能要 1-2 秒，emoji 几乎瞬发。

# nanobot_server.py
_active_reactions: dict[str, tuple[str, str]] = {}
# trace_id → (message_id, reaction_id)

跟 _active_cards 一样的追踪机制，shutdown 时也会清理（删除尚未"对应完"的 emoji）。

8按钮交互 · 卡片不只是显示

lark/buttons.py 393 行专门处理按钮回调。常见按钮：

  • ▶ Stop · 中止当前 agent 跑动 —— 路由到 CommandRouter 的 /stop
  • ✓ 确认 / ✗ 跳过 / ⊘ 取消 · 7 步 autopilot 在 Step 3/4 需要人工决策时用
  • 🔄 重试 · 错误卡片提供，重新跑同样输入
  • 🔍 trace · 跳到 agent doctor URL（带 trace_id 的诊断页）
按钮的 "回调地址" 必须固定 —— IM platform 卡片配置里写死 webhook URL，agent 服务重启后路径必须不变。这就是为啥按钮处理走独立的 /button_callback 路径而不是 /webhook。

9评论通道 · 第二种触发方式

lark/comments.py + lark/comment_channel.py 一共 1000+ 行 —— 处理一种特殊的触发：用户在 IM platform 文档里 @Athena 评论。

区别于 IM @：

  • 没有"群对话"概念，每条评论独立上下文
  • 触发 Athena 之后，回复也作为"对评论的回复"挂在原评论下
  • 非常适合"agent 在文档里帮我把这段改一下"这种用法

实现上 comment_channel.py 实现了同一个 ChannelProtocol（在 nanobot 框架里抽象），所以底层 agent loop 完全不关心是 IM 还是评论 —— 只看到统一的 "InboundMessage" 接口。
