04 · 难度路由 + 模型分级

为啥 Athena 一天烧不爆 — 把 80% 流量挡在便宜模型这边
Opus 比 Sonnet 贵约 5×。Athena 一天处理几千条 IM platform 消息，如果每条都 Opus，账单会爆。所以加了个 3 层路由：硬规则 → Bedrock Nova Micro 判官（~$0.000007/call）→ regex 评分兜底。
最终输出 5 个类别 → 映射到 1-5 难度 → 选 Sonnet 或 Opus。
命名坑： 文件叫 gemini_judge.py，但代码里 JUDGE_MODEL = "amazon.nova-micro-v1:0" —— 历史命名遗留，实际用的不是 Gemini，是 Bedrock Nova Micro。复用主 Bedrock 凭证，不用额外 API key。

1三层路由总览

Layer 1 硬规则 · hard_rule_route() ~0ms · $0
几个 if-else，命中就立刻返回，不调任何 LLM。覆盖 30-50% 的"明显能判"的消息。
↓ return None (没命中)
Layer 2 Nova Micro 判官 · judge_difficulty() <1s · $0.000007/call
Bedrock amazon.nova-micro-v1:0 模型当裁判，给它一个分类 prompt。便宜到可以每条消息都跑一次。
feature flag SMART_ROUTING_V2=true（默认开）。设 false 就跳过这层直接 V1。
↓ except / timeout
Layer 3 V1 启发式 · _v1_heuristic() ~0ms · $0
老版本 regex 评分逻辑，绝不挂。Layer 2 网络故障 / Nova Micro 抽风 / 模型返回非法分类时兜底。

三层不管哪层胜出，最终输出都规约到这 5 类，再映射到难度 → 模型：

4Layer 2 · Nova Micro 判官 prompt

核心 prompt（截选关键部分）：

# JUDGE_PROMPT
You are a request classifier. Classify the user's request into exactly ONE category.

Categories:
- GREETING: trivial greeting, acknowledgment, thanks, affirmation
- SIMPLE: factual lookup, short answer, quick question, translation, status check
- CODING: write/edit/debug/review code, build UI, modify files, DevOps
- ANALYSIS: data analysis, research, investigation, multi-step reasoning, comparison
- CREATIVE: long-form writing, copywriting, content creation, brainstorming

Decision rules (IMPORTANT — follow strictly):
1. If user asks to WRITE, EDIT, FIX, BUILD, MODIFY code/files/UI → CODING
2. If user asks to ANALYZE, RESEARCH, INVESTIGATE, COMPARE, PLAN → ANALYSIS
3. ... (more rules)

Chinese examples:
- "帮我改一下这个页面" → CODING
- "帮我查一下这个数据" → SIMPLE
- "分析一下竞品策略" → ANALYSIS
- "写个脚本" → CODING

Context: {context}
User message: {text}

Reply with ONLY the category name (one word, uppercase). Nothing else.

判官的几个设计要点：

  • 中英文混合 examples —— Athena 用户群体中文为主，明确举中文例子比纯英文准
  • "Short ≠ simple" rule —— 短消息也可能是 CODING（"帮我改一下" 才 6 字符）
  • 用户显式要 opus → ANALYSIS（"用 opus 帮我…"）—— 尊重用户升级意图
  • text 截断到 1500 字符 —— 长消息不全喂给判官，省 token
  • VALID_CATEGORIES 校验 —— 判官返回非法分类时，调用方丢弃，走 V1

6模型选择 · 难度 → Bedrock model_id

路由完产出 difficulty: int，由 MODEL_TIERS 字典映射到具体 Bedrock model ID：

MODEL_TIERS = {
    1: "us.anthropic.claude-sonnet-4-6",
    2: "us.anthropic.claude-sonnet-4-6",
    3: "us.anthropic.claude-sonnet-4-6",
    4: "us.anthropic.claude-opus-4-6-v1",
    5: "us.anthropic.claude-opus-4-6-v1",
}

具体选模型在 bedrock_provider.py 里用 contextvar per-request 注入。这样同一个 provider 实例能为不同请求选不同模型，不污染全局状态。

这是"Bedrock fallback chain" 之外的事：
  1. 难度路由 → 选 Sonnet / Opus
  2. Bedrock us-east-1 primary
  3. 503/throttle → fallback to us-west-2
  4. 流式 stall → fallback to VectorDB / OpenRouter
路由层选模型 = "聪明地省钱"，fallback 层 = "扛住故障"。两件事独立。

7完整调用 · 一次消息走过的路

# nanobot_server.py / server.py 调度时
difficulty, category = await assess_difficulty_async(
    text=user_msg,
    has_image=event.has_image,
    has_file=event.has_file,
)
# difficulty 是 1-5 整数, category 是 5 类之一字符串

# 决定本次用哪个模型 → 设置 contextvar
model_id = MODEL_TIERS[difficulty]
with bedrock_model_override(model_id):
    result = await agent_loop.handle_message(user_msg)
# 这一句之后整个 agent loop 所有 LLM 调用都用这个 model

路由只在请求入口判定一次，整个 agent loop 期间不会换模型。同一个 conversation 不同 turn 可以用不同模型（"帮我改下页面" → Opus；后续 "看好了" → Sonnet）。

9UsageMeter · 路由完了记账

每次 LLM 调用（无论判官还是主力）都过 UsageMeter 记账：

  • prompt_tokens / completion_tokens —— 算钱
  • cache_read_input_tokens —— prompt cache 命中的部分（Anthropic 给打折）
  • cache_creation_input_tokens —— 第一次创建 cache 的 token（贵 25%）
  • fallback_context —— 走 fallback chain 时记到哪一级，分别统计成本

这些 metric 喂给每日 IM platform cost report (amy_cost_daily_report.py)，团队能看到"Nova Micro 判官花了多少 / 主 Sonnet 花了多少 / Opus 花了多少"。

反馈环： 看到 "今天 80% 流量走 Sonnet 但成本仍 60% 在 Opus" → 知道 Opus 流量单价占主导 → 优化路由阈值 / prompt 设计 → 下次复盘看效果。路由不是配一次就完，是持续优化。
