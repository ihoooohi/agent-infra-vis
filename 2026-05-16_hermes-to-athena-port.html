PART 01

原文究竟在说什么

读原文先抓住关键转折，否则会把"反向论证"读成"正向蓝图"。

《如果 Hermes 给团队用——从 SQLite 到 Postgres 的架构推演》通篇结构是： Hermes 单机版的所有设计都是为"每个人在自己机器上跑自己的 agent"优化的——SQLite + 本地 JSONL + 单进程； 团队化会撞四堵墙（写并发、跨机器、权限隔离、容量），所以底座要换：SQLite→Postgres+RLS、 FTS5→Meilisearch/OpenSearch、本地 JSONL→S3 分层、+ Redis 锁/限流、+ Vault 密钥池。

但作者收尾给了一个关键判断（这是原文最容易被忽略的一句）：

所以 Hermes 团队版不会是"加几个配置"，而会是另一个产品——共享 Hermes 的 prompt 工程、技能系统、curator/GEPA 这些与存储无关的核心，但底座完全重建。

这句话直接决定了"移植"的合法路径：底座（SQLite/JSONL/Postgres/S3）不需要移植——Athena 已经有自己的多租户底座（Bedrock + VectorDB pgvector + 共享 workspace + Hatchet 持久任务）。
真正能搬的只有作者自己点名的三块"与存储无关的核心"。下表是这三块的精确定位：

把这张表抓住，就知道"移植 Hermes 到 Athena" 不是搬代码——Athena 几乎全有了——而是查对应件、补缺、把没接通的连线接好。

PART 02

Athena 的真实形态对照

PART 03

核心发现 · 没接通的连线

这是整篇报告最有价值的一段——决定了"小工作量移植"到底有多小。
FINDING · Dream 子系统线上不工作的根因

事实链：

  1. nanobot/agent/memory.py 共 785 行，定义了 MemoryStore / Consolidator / Dream 三个完整类。
  2. nanobot/agent/loop.py:312-326 AgentLoop.__init__ 总是实例化 self.consolidator 和 self.dream——这部分没争议。
  3. Consolidator 在 loop.py:629/647/665/706 通过 maybe_consolidate_by_tokens 自动 触发——session 内归档老消息进 history.jsonl。这一半是活的。
  4. Dream 是独立的 system job，必须靠 CronService.register_system_job(id="dream", ...) 注册才会跑。
  5. nanobot/cli/commands.py:805-818（CLI 入口）有这段注册： 
    cron.register_system_job(CronJob(
    id="dream",
    name="dream",
    schedule=dream_cfg.build_schedule(config.agents.defaults.timezone),
    payload=CronPayload(kind="system_event"),
))
  6. nanobot_server.py（fleet 实际启动文件，229976 字节）整文件没有 cron、CronService、register_system_job、Dream 字样——只把 AgentLoop 起来，但没起 cron service，也没注册 Dream job。
  7. 项目 CLAUDE.md 同时存在一条互相矛盾的标注："workspace has no .git, .cursor/.dream_cursor haven't moved for weeks" / "Compaction Hook replaced them"——前半句是事实（Dream 没跑），后半句是误判（Compaction Hook 解决会话内压缩 ≠ Dream 跨会话沉淀）。
  8. 但 nanobot/skills/memory/SKILL.md 仍然告诉 agent："SOUL/USER/MEMORY 由 Dream 自动管理，请勿编辑"——这条对 Hermes/CLI 路径是对的，对 fleet 部署是失实承诺。

含义：Athena 线上的 SOUL/USER/MEMORY 处于"agent 以为自动维护，实际没人维护"的状态。MEMORY.md 顶上那行 Last compressed: YYYY-MM-DD 是手动加的人类时间戳，没有自动 Dream 修订。

FINDING · 共享 workspace 决定了 Dream 不能裸接通

CLAUDE.md 提示的硬约束：13 个 fleet 容器 mount 同一份 /home/ubuntu/agent-data/prod/workspace/。如果直接像 Hermes 那样在每个 bot 里跑 Dream：

  • 13 个进程并发改 workspace/SOUL.md、USER.md、memory/MEMORY.md → 写覆盖、内容竞争
  • Dream 的 GitStore.auto_commit 同时 fire 13 路 → git index lock 冲突
  • 这是单机版搬到团队版第一个会撞的墙——但原文没讲这种"伪团队版"（同一文件系统多进程），它直接跳到"换 Postgres"

所以 Athena 的正确接通方式是：选一个 leader bot（比如 Athena 本体）跑 Dream，其他 skin-bot 不跑 Dream。具体策略在 Part 4 的 S1 方案里。

PART 04

移植方案 · 三档工作量

按"投入 / 收益 / 风险"排序。强烈建议只做 S 档，必要时做 M1，剩下都不值。
S1 · 强烈推荐 ≈ 50–80 行 · 1 PR · 0.5 工时
把 Dream system job 接到 fleet 启动路径
把 cli/commands.py:805-818 那段 cron 注册搬到 nanobot_server.py，但加 leader 选举——只在 Athena 本体跑，skin-bot 跳过。

做什么：

  • 在 nanobot_server.py 启动时按 BOT_NAME == "athena"（或 fleet.json 里加 "dream_leader": true 标志）判断当前是否为 leader

  • 非 leader 仍然有 self.consolidator（session 内归档），但 Dream 不触发——他们写的 history.jsonl 会被 leader 统一消化
  • 同步更新 CLAUDE.md 里那条"Dead code, Don't try to enable"标注——它已经过时
  • Dream 跑通后，nanobot/skills/memory/SKILL.md 的承诺就变成真的了
~ agent/nanobot_server.py   加 CronService init + Dream 注册（仿 cli/commands.py:790-820）
~ agent/nanobot-fleet.json   Athena 那条加 "dream_leader": true
~ agent/CLAUDE.md   修订 "Dead code in nanobot/agent/memory.py" 段落
+ agent/tests/test_dream_leader.py   验证只有 athena 注册 Dream job
~ agent/scripts/check_dream_status.sh   加 leader 上 .dream_cursor mtime 健康检查

风险点：Dream Phase 2 用 AgentRunner 调用 LLM 增量编辑文件——leader 选举 + 文件锁要确保 Phase 2 期间没人编辑同一文件。MemoryStore 已有 self._git（GitStore）做版本化，但 workspace 没初始化 git，建议不开 GitStore（避免 13 容器锁冲突），改成 Dream 跑前对 SOUL/USER/MEMORY 做 .bak 副本（rollback 用 /dream-restore 命令——这命令已经存在）。

在 MEMORY.md/SOUL.md/USER.md 顶部加结构化 frontmatter
借 Hermes 文章里"三层 SOUL.md"思路的最小版本——不分层，但在文件头声明 owner / last_updated / version。

M1 · 可考虑 ≈ 200–300 行 · 2–3 PR · 1–2 工时
三层 SOUL.md：Org / Team / User
原文专门画了一张图（"Org SOUL → Team SOUL → User SOUL → 最终 system prompt"）。Athena 已有"全 fleet 共享 SOUL.md + 各 skin 的 skins/<bot>/"两层，缺最上面的 Org 层。

做什么：

  • 新增 workspace/ORG_SOUL.md —— 公司级 guardrail（不准外泄源码、合规禁词等）

  • skins/<bot>/SOUL.md 视为 User/Persona 层（各 skin 的个性）

收益 / 风险评估：收益是 governance（Org 层可独立审批合规规则），但 Athena 现在没业务需求要这层——目前所有 fleet bot 都受同一个公司管。建议只在出现"Athena 平台开始接外部租户"时才做。

~ agent/nanobot/agent/memory.py   MemoryStore 加 read_org_soul()

M2 · 可考虑 ≈ 150 行 · 2 PR · 1 工时
用 VectorDB pg_trgm 给 history.jsonl 加真 trigram FTS
Hermes 用 SQLite FTS5 做中文 substring 检索；Athena 现在靠 agent 主动调 grep 凑合。当 history shard 多到一定量级（几千文件），grep 会慢、LLM 也猜不到搜什么。

做什么：VectorDB 是 Postgres + pgvector，建一张 history_search 表，content text + agent_id text + ts timestamptz，建 gin_trgm_ops 索引。Consolidator 写 jsonl 时同步 insert 一条；新增 history_grep(query) 工具给 agent。

不做这个的理由：当前共享 workspace 量级（百级 daily shard），ripgrep 几十毫秒搞定，不构成瓶颈。原文也提到这是 50 用户以上才必要。Athena 13 个 bot 共享一个 workspace ≈ 等效 1 个用户的存储量。

L · 不建议 ≈ 1000+ 行 · 6+ PR · 1–2 周
把 memory backend 抽象成 plugin（honcho / mem0 / supermemory）
原文最后提到 Hermes 的 plugin 系统有 memory plugin。但这是为"个人用户自由选记忆服务"设计的——Athena 是 SaaS 后台，只用一个 VectorDB backend，没有第二个 backend 候选。

为什么不做：

  • YAGNI：Athena 不需要换 backend，VectorDB 跑得好

  • 真正需要替换 backend 时再做不晚——届时业务方向已明朗，抽象点也更准
L · 不建议 不可量化
完整复刻 Hermes 单机版（SQLite + JSONL）作为"个人用户产品线"
即"用 Athena 的代码骨架重新搭一个面向开发者本地用的 Hermes"。这是产品决策不是工程决策。

原文已经说了——这本身就是另一个产品。如果真要做，应该 fork Hermes 不是改 Athena。

PART 05

风险点与执行顺序

动手前必读，特别是关于共享 workspace 的并发约束。
RISK · workspace 共享写入冲突
Athena 的 workspace mount 给 13 个容器。任何"激活 Dream"的动作必须前置leader 选举，否则当 Dream Phase 2 用 LLM 增量改 MEMORY.md 时，其他 bot 在并行追加 memory/YYYY-MM-DD.md daily notes——文件不冲突，但 LLM 的"全局事实判断"会基于不完整的 snapshot。**必须做的最小化**：Dream 跑之前 fsync + 读时校验 mtime，发现期间被改过就跳过本轮，下一轮再试。
RISK · 文档与代码失真
当前 nanobot/skills/memory/SKILL.md 告诉 agent "SOUL 由 Dream 自动管理 / 请勿编辑"，但实际 Dream 没跑。若 Dream 接通前发布报告或开放 agent 编辑，agent 会按 SKILL.md 的指示不去维护 SOUL.md——结果谁都没改。S1 接通 Dream 前，建议先把 SKILL.md 改成 "如果你看到 SOUL/USER 内容明显过期/错误，请用 edit_file 直接修正"，把 fallback 路径打开。
RISK · Hermes 的 RLS 论点不适用 Athena
原文花了很多篇幅讲 Postgres RLS（按 team_id 行级隔离）。Athena 不是这种产品形态——所有 fleet bot 都为同一个 The Company 团队服务，没有外部租户。已有的 Python 后置 agent_id == AGENT_ID 过滤足够。不要误读原文然后给 VectorDB 加一层根本用不上的 RLS——抽象成本远大于收益。
RISK · Dream 的 LLM 成本
Dream Phase 1 + Phase 2 都调 LLM，每次 Phase 2 是 multi-iteration AgentRunner（最多 10 轮，每轮可能 16K chars 工具输出）。必须配 LLM 模型 + 频率上限：用 Sonnet 而不是 Opus、每日最多 1 次、单次 batch 上限 20 entries（已是 Dream.max_batch_size 默认值）。先在 dev fleet 跑一周观测成本再上线。
