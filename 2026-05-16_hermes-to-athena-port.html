关键差异：Hermes 是"一文件夹全包"，Athena 是"一份共享 workspace + N 个容器 + 一个中央向量库"。这套结构决定了"接 Dream"必须做 leader 选举。

PART 05

风险点 + 执行时间轴

动手前必读这 4 个风险，特别第一个。
! WORKSPACE 写并发
13 容器 mount 同一 workspace · 必须 leader 选举，否则 Dream Phase2 并发改 SOUL.md 出现写覆盖。最小防御：跑前 fsync + 读 mtime 校验。
! 文档失真
SKILL.md 当前承诺"Dream 自动管理 / 请勿编辑"。S1 接通前先把 SKILL.md 改成允许 agent 主动修订过期内容——打开 fallback 路径，免得"谁都没改"。
! RLS 论点不适用
原文花大篇幅讲 Postgres RLS。Athena 不是这种产品形态——所有 fleet bot 为同一团队服务，无外部租户。已有的 Python 后置 agent_id 过滤足够，不要画蛇添足。
! Dream LLM 成本
Dream Phase1+Phase2 都调 LLM。必须设 Sonnet (非 Opus) + 每日 1 次 + batch 上限 20。先在 dev 容器跑一周观测再上线。
执行时间轴 · 推荐顺序
推荐节奏：第一个 PR 做 1+2+3+4 (≈ 0.9 工时)，观察 2 周；M1/M2 只在有触发条件时再做。
整篇结论：原文不是"团队版蓝图"——它是"反证团队版需要换底座"。Athena 已经踩在团队侧的山顶。从这个山顶往回看 Hermes，能搬的只剩三块「与存储无关」的核心。Athena 这边①② 全部齐活，③ 的代码（Consolidator + Dream 共 600 行）也在仓库里，只是 fleet 启动路径漏接了 cron 注册——加 leader 选举 + 注册 cron job ≈ 不到 100 行就把"自演化"真正接通。其它 M/L 全是触发驱动，无业务推不做。
