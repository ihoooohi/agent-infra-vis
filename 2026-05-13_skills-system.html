03 · Skills 动态加载系统

SKILL.md 是 agent 的"工具说明书"—— 不是代码，是写给 LLM 看的
Skill ≠ Tool。Tool 是 Python 函数（agent 能调用），Skill 是 Markdown 文档（教 agent 如何用工具完成某类任务）。每个 skill 就是一个目录 + 一份 SKILL.md，带 YAML frontmatter + 自然语言说明。
Athena / Echo / 14 个 skin 共享一套 9 个 builtin skill，每个 skin 也可以在自己 workspace 里再装专属 skill 覆盖 / 扩展。

1Skill vs Tool · 别搞混

Tool
Python 函数注册到 ToolRegistry。LLM 用 tool_use 调用，参数 schema 由 Tool 类定义。
@register_tool class GrepTool(Tool): name = "grep" schema = {...} async def execute(self, args): ...
Skill
Markdown 文档，frontmatter 配元数据。塞进 system prompt（或被 agent 主动读取），告诉 agent "在 X 场景下怎么用 grep / gh / brew"。
# SKILL.md --- name: github description: ... metadata: {"nanobot":{...}} --- # 怎么用 gh CLI 查 PR ...
简单类比： Tool 是螺丝刀，Skill 是说明书。LLM 拿到螺丝刀也能瞎拧，但有说明书就知道"拧这个螺丝要先松那个垫片"。

2两种 skill 源 · workspace 覆盖 builtin

SkillsLoader 找 skill 时按固定顺序，workspace 优先 —— 同名 skill 可以"覆盖"框架版本。

workspace skills
{workspace}/skills/{name}/SKILL.md
  • 每个 agent 自己的 skin 目录里
  • 优先级最高（同名覆盖 builtin）
  • 可以放 _archived/ 子目录"封存"—— 不主动加载
  • 实测：Athena / Echo / 14 skin 目前都没自定义 skill
builtin skills
nanobot/skills/{name}/SKILL.md
  • nanobot-ai 框架自带的 9 个
  • 所有 agent 默认共享
  • 改它要发 nanobot-ai 新版 + 重建镜像
  • 已实测：13 fleet agent 全都依赖这套

4SKILL.md 格式 · 拆 github 这一份

---
name: github
description: "Interact with GitHub using the `gh` CLI..."
metadata: {"nanobot": {
    "emoji": "🐙",
    "requires": {"bins": ["gh"]},
    "install": [
      {"id":"brew","kind":"brew","formula":"gh","bins":["gh"]},
      {"id":"apt","kind":"apt","package":"gh"}
    ]
}}
---

# GitHub Skill
Use the `gh` CLI to interact with GitHub...

## Pull Requests
Check CI status on a PR:
```bash
gh pr checks 55 --repo owner/repo
```
...

frontmatter 的关键字段：

字段作用
nameskill 标识，需匹配目录名
description一句话总结，给 agent 看到清单时决定要不要 load
metadata.nanobot.requires.bins需要哪些 CLI 工具（如 gh），运行时 shutil.which() 检查
metadata.nanobot.requires.env需要哪些环境变量（如 GITHUB_TOKEN），os.environ.get() 检查
metadata.nanobot.install安装指引（按 brew / apt / pip 分），可被前端用来一键装
metadata.nanobot.alwaystrue 则每次对话都自动注入到 system prompt
metadata.nanobot.emojiUI 显示用

5两种加载模式 · always vs 渐进式

Always 模式
SKILL.md 标 always: true。每次 system prompt 构造时整段注入。
适合：每对话都用得到的核心能力（比如 Athena 的"如何用 IM 卡片"）。
build_system_prompt(): parts = [identity, bootstrap, memory] always_skills = skills.get_always_skills() if always_skills: parts.append(skills.load_skills_for_context(always_skills)) parts.append(skills_summary) # 其他 skill 只列摘要 return "\n\n---\n\n".join(parts)
渐进式加载
默认只把 skill 名字 + 描述放进 system prompt（XML 列表）。agent 要用某个时，用 read_file 主动读 SKILL.md。
适合：长尾能力，省 context token。
<skills> <skill available="true"> <name>github</name> <description>Interact with GitHub...</description> <location>.../github/SKILL.md</location> </skill> </skills>
渐进式加载的省钱算盘： 9 个 skill 全文塞进每个 prompt = 数千 token 浪费。只塞清单 = 几百 token。当 agent 真要用 github 时它会自己 read_file(.../github/SKILL.md) —— 按需付费。

6

7skills_summary 长啥样 · XML 给 LLM 看

这是 agent 看到的"能力清单"，LLM 会根据 description 决定要不要去深读：

<skills> <skill available="true"> <name>github</name> <description>Interact with GitHub using `gh` CLI...</description> <location>/usr/local/lib/.../skills/github/SKILL.md</location> </skill> <skill available="false"> <name>tmux</name> <description>Remote-control tmux sessions</description> <location>.../skills/tmux/SKILL.md</location> <requires>CLI: tmux</requires> </skill> <!-- archived: name-only, read SKILL.md to reactivate --> <skill archived="true"><name>old-experiment</name></skill> </skills>
archived 的 skill 只列名字 · 不占 token： 放在 workspace/skills/_archived/ 子目录的 skill 只输出 name，不读取 description / location / 检查 requires。agent 想用得自己手动 read SKILL.md 把它"激活"。

8实测 · 14 个 skin 都用啥 skill

线上 skin 目录（host: ）： fin-bot · skin-05 · skin-06 · skin-11 · skin-08 · skin-01 skin-04 · skin-12 · skin-09 · skin-10 · skin-03 · skin-03b skin-07 · skin-02 （共 14 个 skin · 加上 Athena 主 = 15 个角色，fin-bot 是 Athena 的金融分支）

每个 skin 目录里只有 4 个文件 —— 没有 workspace skills/。也就是说 14 个 skin 全靠 builtin 9 个 skill：

AGENTS.md
IDENTITY.md
SOUL.md
USER.md
架构观察： Skill 设计上支持 per-agent 定制（workspace 优先），但实际生产里没有 skin 用到这个能力。改一个 skill 影响所有 agent —— 这是双刃剑：统一维护方便，但任何 skill 错误都全员中招。

9SkillUsageTracker · 记账

每次有 always skill 被注入到 prompt，SkillUsageTracker.record_batch(skills, "always", session_key) 会记一笔。这个数据后续做"哪些 skill 真的常被用 / 哪些没人理"的统计 —— 删冗余 skill 的依据。

反馈环： always skill 注入 → 记账 · 渐进式 skill 被 read_file → 统计 · pgvector memory surfaced_count 也记账。三套机制都在收集"什么真有用"的信号，用于持续精简 context。
