02 · Memory 系统

三段式 · 线上实测 → 代码预计 → 综合评价

7Consolidator · 框架设计的"实时归档"

框架设想：每轮 agent iteration 后，Consolidator.estimate_session_prompt_tokens() 估当前 prompt 大小。超过 context_window 阈值 → 走 archive 流程：

step 1→estimate_session_prompt_tokens(session)framework
step 2→如果超 → tokens_to_remove = current - targetframework
step 3→pick_consolidation_boundary() 找一个 user-turn 边界framework
step 4→LLM 摘要 boundary 之前的对话LLM
step 5→摘要 append 到 memory/history.jsonlfilesystem
step 6→session.last_consolidated 更新framework

8Dream · 框架设计的"离线两阶段"

Dream 是 cron 触发的离线进程，把 history.jsonl 里的对话固化进 MEMORY/SOUL/USER 三个文档。设计上分两阶段：

Phase 1 · 分析
LLM 调用产出 analysis summary
读 .dream_cursor 找未处理的 history (默认每批 20 条)，把历史片段 + 当前 MEMORY/SOUL/USER 全文喂给 LLM，输出"这批对话里有什么新事实 / 偏好 / 该更新到哪里"。
→
Phase 2 · 增量编辑
AgentRunner + EditFile 工具改文件
不让 LLM "重写整个文件"（容易幻觉性删除），开个 mini agent 只给 ReadFile / EditFile 工具，让它一处一处增量编辑。改完 GitStore 自动 commit，git log 能看到每次修改。
设计亮点： Phase 2 用工具增量编辑 vs 整文件重写 —— LLM 全文重写容易"幻觉性删除"重要信息。Edit tool 的 search-and-replace 接口让 LLM 必须精确定位才能改，安全得多。

Dream 的 prompt 写法：

# Phase 1 input
## Conversation History
{history_text}

## Current MEMORY.md
{current_memory}

## Current SOUL.md
{current_soul}

## Current USER.md
{current_user}

# Phase 2 spawns sub-agent with tools=[ReadFileTool, EditFileTool]
# Prompt: "Based on analysis above, edit files where needed"

9ContextBuilder · 框架的注入路径

框架代码里 ContextBuilder 一进来就自动注入 memory 到 system prompt：

def build_system_prompt(self, skill_names=None):
    parts = [self._get_identity()]

    bootstrap = self._load_bootstrap_files()    # AGENTS/SOUL/USER/TOOLS.md
    if bootstrap:
        parts.append(bootstrap)

    memory = self.memory.get_memory_context()   # MemoryStore.read_memory()
    if memory:
        parts.append(f"# Memory\n\n{memory}")

    always_skills = self.skills.get_always_skills()
    if always_skills:
        always_content = self.skills.load_skills_for_context(always_skills)
        if always_content:
            parts.append(f"# Active Skills\n\n{always_content}")

    skills_summary = self.skills.build_skills_summary()
    if skills_summary:
        parts.append(...)

    return "\n\n---\n\n".join(parts)

注意：pgvector 不在这里调用 —— 框架设计是 MemoryStore.read_memory() 读本地 MEMORY.md 文件。pgvector 远程由 MemoryService 单独提供 search() API，调用方需自己决定何时用（agent 通过工具调用或 ContextBuilder 显式注入）。

11为什么会有这么大分歧

一个 well-designed 框架的核心组件居然有 6/10 项是 dead code，背后通常有几个原因：

① 框架是通用，Athena 是专用
nanobot-ai 是"个人 AI 助理框架"通用设计。Athena 是"团队 AI 经理"专用产品。一个"按日 jsonl shard 自动归档"对个人用户够，对 13 agent 共享的组织级 memory 不够 → 自家造五阶段 + 共享 markdown 知识库。
② 框架抽象 vs 业务直觉
"Dream 用 LLM 增量编辑 SOUL/USER/MEMORY"很优雅，但实际Athena 需要的事实更分散（按日 log / 主题文件 / 客户档案 / 用户偏好），三个固定文档装不下。一堆按主题命名的 markdown 更朴素也更准确。
③ 演化历史 · 框架在变
代码里 Consolidator / Dream 类还在，说明早期试过。后来发现工程上不顺手 → 写了 hooks/compaction.py 取代。
原 class 没删（其他下游用户可能在用），但 Athena 没接进自己的 nanobot_server.py。
④ "把 LLM 当工具调用方"vs"框架内嵌"
框架想"memory 是 system prompt 一部分，自动注入"。实际 Athena 选了把 memory_search 当工具给 agent —— LLM 自己决定何时用，节省 token，灵活性更高。

12教训与建议

🔍
看代码 ≠ 看线上
"框架代码定义了 Class X"不等于"线上跑着 X"。下笔讲架构前必须 SSM ls / docker logs / cursor 文件 mtime 三件套验证。dead code 在大框架里太常见。
📅
cursor 文件是 dead-code 检测器
.cursor / .dream_cursor / .progress 这种"进度记录"文件，如果几周不动 → 对应机制可能已废弃。比逐行 grep 调用方更可靠。
🗑️
删 dead code · 或明确标 "deprecated"
建议团队走查一遍 nanobot/agent/memory.py 的 Consolidator / Dream / GitStore，明确标 dead 或彻底删除。否则未来同事还会继续掉同样的坑。
📐
CLAUDE.md 同步更新
CLAUDE.md 在写"Memory Service (memory.py): VectorDB (pgvector + Gemini, 3072-dim). Dedup: cosine > 0.9 → merge. Staleness: >1 day"。但compaction 五阶段一字不提。建议补充"Compaction Hook is what actually keeps long sessions running"。
🔐
VectorDB agent_id 在服务端隔离
代码两次注释明确："VectorDB match_memories ignores p_agent_id — we post-filter in Python."
这是一个单点失败风险：每次新 search 入口都必须记得手动过滤。建议在 VectorDB SQL 函数加 WHERE agent_id = p_agent_id 服务端强制。
🤝
共享 markdown 知识库 = 组织级 memory
这个设计虽然朴素但很有效 —— 13 agent 共享同一份 MEMORY.md 形成"组织记忆"。代价是需要 namespace 协议（AGENTS.md）+ 信任 agent 自我约束。值得团队继续投入。
一句话总结： Athena 的 memory 在线上是"共享知识库 + VectorDB + compaction"三件套，跟 nanobot 框架的"MemoryStore + Consolidator + Dream" 已经事实上分叉了。前者朴素实用，后者优雅但 dead。理解 Athena 必须看前者；维护 nanobot 框架仍要看后者。
