01 · Agent Loop 核心循环

Athena / Echo / 13 fleet agents 共用的大脑引擎 —— nanobot/agent/loop.py
Agent Loop 是个独立的 PyPI 包 nanobot-ai。它不知道 IM platform、不知道 Athena、不知道营销 —— 它只懂"构造 prompt → 调 LLM → 解析 tool_use → 跑工具 → 循环"。所有产品语义都靠 hooks 和工具注入。
Athena 和 11 个 skin-bot、Echo，都是同一个 AgentLoop 类，靠 workspace 路径 + 注入的 hooks 区分人格。
agent-primary 容器内 pip show nanobot-ai:

4Hooks · 6 个生命周期挂载点

AgentHook 是空的基类，子类实现想关心的方法。_LoopHookChain 把主 hook 和外部传入的 extra_hooks 串起来。所有 hook 调用都有异常隔离：一个 hook 抛错不会挂掉整个 loop。

Hook
用途
何时调用
before_iteration
每轮开始；_LoopHook 在这记起始时间用于算 TTFT
iter 开始
on_stream
每个流式 chunk；_LoopHook 计算首 chunk → TTFT，转发给 IM 卡片 streaming 显示
流式中
on_stream_end
流结束。resuming=True 说明后面还有 tool 调用（卡片转 "working"）；False 说明 final
流结束
before_execute_tools
工具执行前；_LoopHook 把 thought + tool_hint 发到 IM 卡片（"我在调 web_search…"）
解析出 tool_calls 后
after_iteration
每轮结束；_LoopHook 记录 ContextMetrics（TTFT/tokens/tool_calls）+ ctx_loss 检测
iter 结束
finalize_content
最终内容处理 pipeline；_LoopHook 在这去掉 <think> 块；无异常隔离（bug 要暴露）
最后
设计巧思 · finalize_content 故意不做异常隔离： 其他 hook 失败不能挂 loop（log 一下继续），但 finalize 是最终输出的把关，里面 bug 必须让人看到。这种"差异化容错"贯穿整个框架。

6Checkpoint · 容器挂了能续

每轮关键点都打 checkpoint，存到 session 里。容器重启时 SessionManager 加载 session，从 checkpoint 继续 —— 长时间 agent 任务（比如 7 步 autopilot）才不会从头来过。

awaiting_tools
LLM 给出 tool_calls 但还没执行 · 存 assistant_message + pending_tool_calls
→ 重启时知道"上次 LLM 想调什么工具"，可以重跑工具或者跳过
post_tools
工具执行完，准备下一轮 · 存 completed_tool_results
→ 重启时直接进入下一轮 LLM 调用，不重跑工具
completed
final_content 已生成 · 写入 session
→ 重启发现是 completed，直接返回结果

7并发 · 一个进程最多跑几个会话

一个 nanobot 容器进程内有可能同时被多个 chat_id 请求。靠 Semaphore 控制总并发：

env var
NANOBOT_MAX_CONCURRENT_REQUESTS = 8
→
asyncio.Semaphore(8)
→
第 9 个会话排队等
<=0 表示不限。Athena 默认 8，但内部代码里也见过 3 这种保守值。这层 gate 是为了别让一个 LLM 慢请求堵住整个 nanobot。

容器里 nanobot 是 pip 安装的 v0.1.5.post1 the company fork，不是 editable 装的 — 这意味着改仓库的 nanobot/ 源码不会立即生效，需要重新发包 + 重建镜像。但是 auto-agent/ 仓库代码（业务层）是直接 mount 进容器的，改了重启就生效。

已知风险（CLAUDE.md 列出的）： nanobot-ai PyPI 包没有 pin 版本，上游有变更可能静默挂掉。重建镜像前最好先 pip freeze | grep nanobot 锁定版本。
