08 · Fleet 部署架构

一份镜像、一份代码、13 个独立人格 —— 配置驱动的"分身术"
Fleet 的核心想法：一份代码 + 一个 Docker 镜像（the-platform/nanobot-agent），通过 env vars 和挂载不同 workspace 目录跑出 13 个完全独立的"人格"。
配置都在 nanobot-fleet.json，一个文件就是 fleet 全貌的 ground truth —— 谁、用哪个 IM platform app、什么 tier、跑没跑起来。

"分身"的本质： 容器都是同一镜像启动，per-bot env vars 决定身份：AGENT_NAME=skin-07 · LARK_APP_ID= · PORT=3118 · WORKSPACE_PATH=/root/.nanobot/workspace。容器内一启动就根据这些拼出自己的人格（读 agent-skins-data/skins/skin-07/）。

2nanobot-fleet.json · ground truth

整个 fleet 的配置就在这一个 JSON 文件：

{
  "version": 1,
  "metadata": {
    "description": "Nanobot fleet — personal AI agents on nanobot + Hatchet",
    "ec2_instance": "",
    "image": "the-platform/nanobot-agent:latest",
    "server": "auto-agent/nanobot_server.py"
  },
  "defaults": {
    "llm_model": "us.anthropic.claude-opus-4-6-v1",
    "context_window": 618000,
    "max_iterations": 100,
    "shared_workspace": "/home/ubuntu/agent-data/prod/workspace",
    "identity_source": "/home/ubuntu/agent-skins-data/prod/skins",
    "VectorDB_url": "https://e4ii3f8e.us-east.tenant-04.app"
  },
  "port_offset": 100,    // nanobot_port = openclaw_port + 100
  "bots": [
    { "name": "skin-07", "app_id": "cli_a921...",
      "openclaw_port": 3018, "nanobot_port": 3118,
      "tier": "member", "status": "running", ... },
    ... 12 more
  ]
}

关键字段含义：

  • defaults · 所有 bot 共享的默认值（模型、context 窗口、workspace 路径），bot 内可单独 override
  • port_offset = 100 · 每个 bot 跑两套（OpenClaw 老栈 + nanobot 新栈），端口差 100，方便心算转换
  • bots[].app_id · IM 应用 ID，Webhook 路由就靠它区分
  • bots[].open_id · 用户视角的 bot IM platform UID，方便@
  • bots[].tier · senior / member / functional，可能影响 UI 显示或权限（代码层面待挖）
  • bots[].status · running / pending —— pending 是已配置但尚未拉起容器

5每个 bot 的差异化 · 三层 override

同一镜像怎么变出不同人格？三层叠加：

叠加顺序 · 上层覆盖下层
defaults (fleet.json)
llm_model · context_window · workspace 路径 · VectorDB_url · max_iterations=100
per-bot (fleet.json bots[i])
name · app_id · 端口 · tier · 可选 override llm_model（如 skin-01 用 codex）
runtime · skin files
挂载 /home/ubuntu/agent-skins-data/prod/skins/{name}/ → 容器内 SOUL.md/IDENTITY.md/USER.md/AGENTS.md 决定人格

6Volume 挂载 · 共享 vs 隔离

每个 bot 容器有两个关键 mount，一个共享一个隔离：

host (EC2)
/home/ubuntu/agent-data/prod/workspace/
所有 bot 共享 —— skills / 共用文件区 / MEMORY.md 等
→
container
/root/.nanobot/workspace
共用：MEMORY.md / SOUL.md 全 fleet 一样
host (EC2)
/home/ubuntu/agent-skins-data/prod/skins/{bot_name}/
per-bot 隔离 —— 每个 skin 自己一套 4 个 markdown
→
container
/root/.nanobot/identity
独有：每个 bot 的 IDENTITY/SOUL/USER/AGENTS
共享 workspace 的隐患： 13 个 bot 同时读写 /root/.nanobot/workspace —— 如果两个 bot 同时 git commit MEMORY.md 会冲突。memory.py 用 GitStore 包了一层但仍可能 race。实际生产中没爆过，因为 13 个 bot 同时大改的场景极少。

7特殊 bot · skin-01 的"叛逆"

13 个 bot 里 skin-01 是叛徒 —— 它不用 Bedrock Claude：

{
  "name": "skin-01",
  "tier": "senior",
  "llm_model": "openai-codex/gpt-5.5",        // 不是 Bedrock
  "subagent_model": "openai-codex/gpt-5.5",
  "oauth_token_path": "/root/.nanobot/auth/codex.json"
}

对应的容器还跑了一个本地 build 镜像 skin-01-gateway:local（而不是 ECR）—— 这是多 provider 探索的活实验，team 想看不同模型在同样 fleet 框架下表现差异。

架构上的优雅： fleet 框架不关心 llm_model 是 Claude / GPT / Gemini —— provider 抽象在 nanobot/providers/ 下，oauth_token_path 也只是 env override。加新 provider 不动 fleet 代码，只改一个 bot 配置。

8特殊 bot · fin-bot 拒绝迁移

fin-bot 状态是 pending，note 写 "stays on OpenClaw"。表示这个 bot 还在老 TypeScript 栈上跑（agent-fin-prod 容器，agent-gateway 镜像），没打算迁到 nanobot。

原因可能（推测，未确认）：

  • tier 是 "functional" 而不是 senior/member —— 性质不同（功能性 bot，不是分身）
  • fin-bot 跑财务 / cost 相关任务，稳定性比新功能优先级高
  • OpenClaw TS 栈现有逻辑充分够用，迁移 ROI 低

10整 fleet 升级 · 一行命令换全员脑子

fleet 设计最大的运维价值是原子升级：

  1. auto-agent/ 仓库 commit → CI 构建 the-platform/nanobot-agent:{new_hash} 推 ECR
  2. fleet 部署脚本：遍历 13 bot，docker pull 新版 → docker stop && docker rm && docker run
  3. 每个 bot 用同一新镜像启动 —— 7 小时内升级全员（实测 17:00 部署版本 7h 前完成）
  4. 问题模式：旧版本部署崩（如 5/11 那次 11 个 Restarting），从 ECR 重 pull 修好
横向对比 Cetus： Cetus 那边每个租户一个独立镜像 hash（22 个不同），滚动升级更复杂。Agent fleet 因为是"自家分身"（不是付费客户），可以全员同步升级 —— 简单，快，但 blast radius 大。
