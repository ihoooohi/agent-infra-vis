24 个根因 · 不是 bug，是架构错配

三个 case 的统一画像 → "上游缺信息时, LLM 不停下问澄清, memory 永远填补找'形式相似的成型答案'复用"。具体 4 个根因：

RC-1 · 注入位置错
Memory 拼到 user message 同一个 role 里 (违反 message protocol)
nanobot_server.py:2063-2164: memory_context 拼在 augmented_text 同一个 role=user 内容里, 内容混着字面 "User:" / "Assistant:" 字符串。LLM 分不清"用户在转述过去对话"vs"对话历史一部分" → 把过去 assistant turn 当可复用模板。
RC-2 · embedding 不敏感
gemini-embedding-001 对参数性 token 几乎无视
cosine("review pr 2620", "review pr 2442") ≈ 0.95+。MIN_SCORE=0.5 在这种 query 下形同虚设。PR 号 / email / URL / UUID 这种"参数化任务" 是 embedding 检索的盲区。
RC-3 · namespace 太粗
所有 chat 用一个 agent_id, p2p 完全无 chat scope
agent_id 只到 agent-primary-primary, p2p 分支根本没 chat scope 隔离。dedup cosine>0.9 还把多条 merge 成一条 → 单 PR 号被洗成"上次 review 的 PR" 这种泛化记忆。
RC-4 · LLM 在不确定时 confabulate
长 context + 模糊 input + tool 失败 → 选 "看起来合理的答案"
38KB system + 5 memories + 大量历史 + 模糊 input + tool 失败 → 模型选 confabulation 而非 ask for clarification。LLM-as-agent 在 long-context 下的典型失效模式。
一句话定性 · 写在 issue 文档结尾： "三个 case 不是三个 bug, 是同一个架构选择的三种症状 —— nanobot 把'过去对话'当'当前上下文'无条件注入。在 LLM-as-chatbot 时代这是合理设计; 在 LLM-as-agent 时代是错的 —— agent 任务有明确的 input/output 契约, memory 应该是 agent 决定何时调用的工具, 不是默认的 prompt 食材。"

PR #2961 cf5c34450 · 2026-05-06 +401 / -9 · 14 单测
Active memory_search tool with chat scope (P0 follow-up to #2741)
把 Athena 的 memory access 从"被动 RAG 自动注入" → 改成"agent 主动 tool 调用"。
核心新增: nanobot/agent/tools/memory_search.py 266 行 + memory.py 加 chat_id_filter 参数和 is_enabled(); default scope='current_chat' 走 metadata.chat_id post-filter (因为 VectorDB RPC 忽略此字段); 同时写 nanobot/skills/memory/SKILL.md 教 LLM 怎么用这个 tool。

PR #2964 c1ba4d215 · 2026-05-06 +408 / -67 · 7 安全 + 210 测
Cold-start memory tier loading (session-aware)
抽出 system_context.py 161 行专门负责 cold-start 时的 memory 拼装 —— 不是"all-in-one prompt", 是按 chat_type 分 tier:
· p2p (DM): base + 全局 MEMORY.md + per-DM USER_MEMORY.md
· group: base only, MEMORY.md 不再泄入群
同时被 Codex review 抓出来加了 7 个 path-traversal 防御测试 + 210 行覆盖测试。
"结构性消除跨 chat 泄漏可能" —— 比 issue 文档 P1-2 建议的 "XML 包装" 更深。MEMORY.md 物理隔离在 DM, 群里看不到。
PR #3039 557359af3 · 2026-05-07 prompt 改造
AGENTS.md "Grep First" anti-fabrication rule
prompt 层补一条 hard rule: agent 在生成"事实性内容"前必须先用 grep / search 验证, 而不是从记忆里编。这是 P0-2 真实性 sentinel 的提示工程版本: 不在代码层硬拦, 在 LLM 输入层引导 ask for clarification 而非 confabulate。
prompt 层比正则更通用 (能覆盖代码 sentinel 漏的 corner case), 但失去硬保证。和 Skin-02 的 #3111 "Scope Discipline" 互补。
PR #2741 3e66f84f9 · · 2026-05-01 passive RAG 关闭
Disable passive RAG injection (collaborator's fix that paved P0-1)
这是我 PR #2961 的前序: 同事 把整个被动 RAG 直接关闭, 拿掉 nanobot_server.py:2212 附近的 _memory.search() 自动注入。我跟上做了 active memory_search tool 让 agent 仍能拿到 memory。

PR #3111 4860eea68 · Skin-02 · 2026-05-09 prompt scope discipline

prompt 层组合拳: the author "Grep First" + Skin-02 "Scope Discipline" + the author "Data Source Map" (#2967) 三条 prompt rules 协同, 覆盖了原 spec 没明说的 confabulation 子场景。
PART III

OpenClaw 对照

"我们走的方向对不对?" —— OpenClaw 用生产系统给出了答案

5OpenClaw memory 架构 · pull + file-first + scope

OpenClaw 是 the company 的 TypeScript AI agent 框架, 走的是 "pull + file-first + per-agent scope" 路线。看完它的 docs 一句话总结: 我们走的方向跟 OpenClaw 完全一致, 它用生产系统印证了这就是正解。

OpenClaw · 默认设计
pull + file-first + scope
  • Pull 模式: memory 不自动注入, agent 主动调 memory_search / memory_get tool
  • Hybrid retrieval: BM25 + vector, default 0.7/0.3 加权; 可加 MMR + 时间衰减
  • Per-agent + per-channel scope: SQLite 文件隔离 + QMD match.keyPrefix (default deny-all except DM)
  • File-first SSOT: markdown 是事实源, 模型只记得写到磁盘的内容
  • Citation footer: 召回片段带 Source: path#line
  • Pre-compaction memory flush (NO_REPLY): 静默 turn 写盘, 不污染 user output
  • Temporal decay: 半衰期 30 天, evergreen 文件豁免
  • memory-wiki 插件: contradiction + freshness tracking
Athena · 修复后状态
subset of OpenClaw approach
  • Pull 模式 ✓ (PR #2961 active memory_search tool)
  • Hybrid retrieval ✗ (P2-2 未实施, 只有 vector)
  • Per-chat scope ✓ (PR #2961 chat_id_filter + #2964 tier 隔离)
  • File-first ✓ (workspace markdown 早就是事实源)
  • Citation footer ✗ (没加, 但 chat_id_filter 已经能让 LLM 知道 scope)
  • Pre-compaction flush △ (hooks/compaction.py 5 阶段做类似事, 但不是 NO_REPLY 模式)
  • Temporal decay ✗ (只有 >1d 静态 caveat, 没 freshness 权重)
  • 真实性 sentinel ✓ (OpenClaw 没有, Athena 原创: identity.md + "Grep First")

Case A · PR review 错号
✓ 结构性消除
passive RAG 关掉后, embedding 不再有机会把旧 PR 号自动塞进 user message。当前轮 user 给 "review pr 2620" → review.js 拿到的就是 2620, 不可能再被覆盖成 2442。
review.js 本身仍无 user-msg 锚定。若 subagent context / task description 别处来源出现错号, 理论上仍可复发, 但不再走 memory 路径。
Case B · 假执行
△ 部分缓解
passive RAG 关闭 + tier 隔离消除了"跨 chat 召回 ''"的路径。但 P0-2 代码层 sentinel 没做, prompt "Grep First" 是软约束 → LLM 在 tool 失败时仍可能编邮箱地址 + 说"已更新"。
软约束依赖 model compliance, 没硬保证。建议补一段 WRITE_VERBS + MUTATION_TOOLS 正则拦截作为硬底。
Case C · 周报糊弄
△ 部分缓解
passive RAG 关闭 → "召回旧周报"路径堵死。但 P0-3 message_type 全覆盖没做, lark/channel.py:181 对 share_chat/card/rich_text 仍 silently return None → 用户发卡片消息根本进不来。
从 "confabulation paste 旧周报" 降级为 "silent drop / 无回复", 没复发但用户体验变差。建议立刻补 message_type 兜底, 至少抽 URL。

12面试常见追问 · 应对

FOLLOW-UP DEFENSE
  • Q: 三个事故怎么定位到同一根因的?
    A: case A 是 review 错号 (参数被替换), B 是假执行 (LLM 编邮箱), C 是糊弄 (paste 旧周报)。表面三种症状, 但 docker logs grep + session jsonl byte-offset 检查后发现都是"过去对话被无条件注入当前 user message"的延伸。不是 3 个 bug 是 1 个架构选择的 3 种症状。
  • Q: 为什么没按 issue 文档的 P0-1 ~ P1-2 顺序做?
    A: 同事 直接 PR #2741 关闭整个 passive RAG 是个更彻底的根治。原 spec 是"在 p2p 分支加 chat scope"打补丁, 但既然能关掉就跳过补丁路线直接做 P2-1 架构性方案。我接力做 active memory_search 让 agent 仍能拿到 memory, tier 隔离让 DM/group context 干净。
  • Q: P0-2 / P0-3 / P1-1 为什么没用代码层做?
    A: P0-2 sentinel / P1-1 review.js 锚定确实没写正则。我选了 prompt 层 ("Grep First" + Skin-02 的 "Scope Discipline") 替代, 因为 prompt 覆盖面比正则广 (能挡 LLM 输出 corner case)。代价是软约束, 没硬保证。如果未来 model compliance 变差, 还要补硬拦截。P0-3 message_type 是单纯的工程债, 我应该立刻补但优先级排在 PR #2961/#2964 之后。
  • Q: 跟 OpenClaw 的设计对比有什么启示?
    A: OpenClaw 走 "pull + file-first + hybrid retrieval + temporal decay" 全套, Athena 现在做完了 pull + file-first + scope, 差最后的 hybrid retrieval (BM25 + vector) 和 temporal decay (半衰期 30 天)。这两件是下一阶段补全的方向, OpenClaw 用生产系统印证了这是正解。
