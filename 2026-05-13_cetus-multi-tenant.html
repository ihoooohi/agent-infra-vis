09 · Cetus 多租户基础设施

22 家外部客户共用一台机器 · DB 是 SSOT · routeCache 是 30s 内存窗
Cetus 是把 Athena 的能力打包卖给外部客户的 SaaS。同一台 EC2、同一个总路由 tenant-gateway，把进来的 IM/Slack webhook按 chat_id 查 DB 路由表，分发到对应租户的 agent 容器。
DB 是 single source of truth · 内存 routeCache 30s 刷一次 —— 是 Cetus 多租户的两根支柱。

2tenant-gateway · "thin" 是关键

主 gateway 的设计目标 = 路由层零业务、零状态。它只做四件事：

1
收 webhook
IM platform / Slack POST 到 /webhook/lark 或 /slack/events
HTTP
2
验签 + 解 chat_id
parseLarkWebhook(body) / parseSlackWebhook(body) → 拿 (platform, chat_id, sender_id, text)
verify
3
查 routeCache
routeCache.get(`${platform}:${chat_id}`) → agent_slug + container port
Map
4
proxy 转发
createProxyToAgent() → http://localhost:18xxx/webhook (透传 body / headers)
proxy
"thin"的回报： 因为只是 routing，tenant-gateway 容器极轻量、可水平扩展、重启秒级。所有业务（LLM 调用、记忆、工具）在租户容器里跑 —— 升级一个租户不影响其他。

3routeCache · 30 秒内存窗

不可能每条消息都查 DB —— routeCache 是 in-memory Map：

let routeCache = new Map();    // 初始空

async function refreshRouteCache() {
  const rows = await pool.query(`
    SELECT platform, chat_id, agent_slug, agent_port, tenant_slug
    FROM chat_routes JOIN agents USING (agent_id)
  `);
  const newCache = new Map();
  for (const r of rows) {
    newCache.set(`${r.platform}:${r.chat_id}`, r);
  }
  routeCache = newCache;    // 原子替换
}

setInterval(refreshRouteCache, 30_000);    // 30 秒刷一次

几个设计要点：

  • 原子替换 · 不是逐条更新 Map，而是构造完整新 Map 再 routeCache = newCache —— 中间状态对外不可见
  • 30s 窗口 · 新增 chat 路由后最多 30s 才生效 —— 是"快得够用、不占 DB"的折中
  • 不持久 · gateway 重启就丢，但 30s 内重建。冷启动可能丢前 30s 消息（实际看 health 等 cache 满才返 200）
  • 不淘汰策略 · 全表加载，因为 chat_routes 一般 < 1000 行

4DB Schema · 6 张表

全部在 tenant-postgres 单实例，schema 在 init.sql：

tenants
客户组织。name / slug / config JSONB。config 里放每家自己的 IM App ID、品牌色等
agents
AI agent 实例。一个 tenant 可以有多个 agent（业务场景：客服 / 数据 / 内容审核 各一个）
chat_routes
(platform, chat_id) → agent。这是 routeCache 的源。同一个 chat 只能绑一个 agent
tenant_skills
per-tenant skill 权限。某 tenant 能用哪些 skill，是否定制版
default_skills
skill 总目录 · is_default 标的 skill 新 tenant 自动获得
file_audit
tenant workspace 文件改动日志 · 用于安全审计 / 调试
没有 RBAC / auth 系统： Admin API 全部在 /api/*，README 直接写 "no auth yet — TODO"。当前靠 Cloudflare Tunnel 限定 IP（tenant-admin.the-company-ai.com 只能 the company 内网访问）。生产 SaaS 要补 auth。

5端口分配 · 确定性哈希

22 个租户容器，每个要绑 webhook 端口。要么手动分（容易冲突 / 难记），要么确定性哈希：

function agentPort(tenantSlug, agentSlug) {
  const h = sha256(`${tenantSlug}:${agentSlug}`);
  return 18800 + (parseInt(h.slice(0, 8), 16) % 200);
}
// 18800-18999 范围，200 个槽位

性质：

  • 同 (tenant, agent) → 同端口 · 重新启动容器端口不变，gateway 不用 reload
  • 200 个槽位 · 22 个用了 11%，碰撞概率近似生日悖论 ~16%（理论上）
  • 实际碰撞处理 · 检测 isPortListening(port) 已占 → 找下一个空槽位（线性探查）
  • 写入 DB · 最终端口在 agents 表 agent_port 列，gateway 查 routeCache 拿真实端口（不每次重哈希）

7JuiceFS · 跨容器持久化

每个 tenant 容器都无状态 —— 数据全在 /jfs/tenants/the-company/{slug}/：

路径
用途
/jfs/tenants/the-company/{slug}/workspace/
tenant 的 nanobot workspace · SOUL.md / MEMORY.md / 自定义 skills
/jfs/tenants/the-company/{slug}/state/openclaw.json
老 OpenClaw 栈的 state · 一些 tenant 仍跑 OpenClaw
/jfs/shared/skills/
所有 tenant 共享的 default skills

JuiceFS 的关键特性：

  • S3 backend · 实际数据在 S3，本机只是缓存 + 元数据
  • PostgreSQL 元数据 · 文件元信息在 PG（同一台 tenant-postgres，但不同库）
  • 容器重启不丢 · 数据落 S3
  • EC2 换机不丢 · 把 JuiceFS 挂载到新机就还在

8Admin API · 一行命令开 tenant

Cetus 设计上鼓励"一行加新 tenant"。/opt/Cetus/infra/deploy-agent.sh 干这事：

$ /opt/Cetus/infra/deploy-agent.sh \
    --client "NewCustomer" --slug "newcust" \
    --agent-slug "newcust-external" \
    --slack-channels "C123,C456"

背后做的事（手动版的步骤）：

1
POST /api/tenants {name, slug}
建 tenants 记录 · 自动 seed default_skills 到 tenant_skills（每个 is_default=true 的 skill）
DB
2
POST /api/agents {tenant_id, slug, name}
建 agents 记录 · 算 hash(tenant:agent) → 分配 port → 写 agent_port
DB
3
创建 JuiceFS 目录
mkdir /jfs/tenants/the-company/newcust/{workspace, state} · 写默认 SOUL.md / openclaw.json
FS
4
docker run --name Cetus-newcust ...
挂 JuiceFS 卷 + 端口 + tenant env → 启动 the-platform/tenant-gateway 容器
docker
5
POST /api/routes (×N)
每个 (slack_channel / lark_chat_id, agent_id) 写 chat_routes —— 30s 后 routeCache 自动 pick up
DB
关键： 加新 tenant 不重启 tenant-gateway —— DB 写完，下次 routeCache 刷新就识别新路由。这是"SSOT + cache"模式的运维红利。

10对比 Agent fleet · 两套理念

维度
Agent fleet vs Cetus
租户单位
Athena: 13 bot "分身"，全是 the company 自家人格
Cetus: 22+ 客户公司，每家可能多个 agent
配置存储
Athena: 一份 nanobot-fleet.json + git
Cetus: PostgreSQL 6 表，Admin API 可改
路由方式
Athena: 每个 bot 自己的 IM App，独立 webhook URL
Cetus: 一个总 gateway + chat_routes 表动态分发
隔离
Athena: 文件级（不同 skin 目录），但共享 workspace
Cetus: JuiceFS 路径隔离 + 容器隔离 + DB 行级隔离
部署
Athena: 一次 docker pull 全员升级
Cetus: 每个租户独立容器，可滚动升级
运行时
Athena: 全 Python nanobot
Cetus: TypeScript gateway + 老 OpenClaw / 新 nanobot 共存（迁移中）
扩展性
Athena: 加新 bot 改 fleet.json + 部署
Cetus: 一行 deploy-agent.sh，运行时加
关键区别： Athena 是静态配置驱动（fleet.json 是 ground truth），Cetus 是DB-driven 动态租户（chat_routes 是 ground truth + 30s cache）。Athena 适合"自家分身"，Cetus 适合"SaaS 业务"。
